## 一、原理简述（不耐心看可跳过）

### 1.1 「接通两条链路」是什么意思

在重构之前，项目里两条扫码链路独立：

```
普通扫码链路 (ScanHub + ScanFlowDialog)
  扫码 → /available-actions → 弹窗手填 fromNode/toNode/省/市
       → POST /api/traces/{code}/events
  缺：不关联运单，每次提交是孤立事件

任务扫码链路 (TraceFlowTaskWorkbench)
  打开运单 → 连续扫码 → POST /api/trace-flow-tasks/{id}/scan
  缺：必须先进特定页面，不能从主扫码工位发起
```

方案 B 把两条链路在 UI 层并接：在 ScanFlowDialog（普通扫码弹窗）里加运单 dropdown，选定任务后自动填字段 + 路由到任务扫码端点；不选任务则走原 events 路径不变。

### 1.2 候选任务怎么算

后端 [`TraceFlowTaskServiceImpl.findCandidateFlowTasksForTrace`](backend/src/main/java/com/example/trace/service/impl/TraceFlowTaskServiceImpl.java) 按以下规则计算候选：

| 任务 type | 当前 status | 节点条件 | 派生 actionType |
|---|---|---|---|
| `OUTBOUND` | IN_STOCK | source 匹配 currentNode | OUTBOUND |
| `OUTBOUND` | IN_TRANSIT/TRANSFERRED | target 匹配 currentNode 且 actualQuantity>0 | INBOUND（目的地接收） |
| `INBOUND` / `RECEIVE` | IN_TRANSIT/TRANSFERRED | target 匹配 currentNode | INBOUND |
| `TRANSFER` | * | * | ❌ 不支持（B18 未实现任务扫码闭环） |

并且 task.status 必须 ∈ {`CREATED`, `PROCESSING`}。

前端再按"用户在 ScanHub 选定的动作类型"二次过滤——例如用户点了「确认出库」(OUTBOUND)，下拉里只展示 compatibleActionType=OUTBOUND 的任务，避免误选。

### 1.3 已修复的省/市 prefill 问题（2026-05-11 手动测试发现）

**症状**：选定 dropdown 后省份字段空白（截图见对话）。

**根因**：
1. seed 脚本把 `trace_node.province` 存成简称 `"江苏"`，前端 [REGIONS catalog](frontend/src/shared/data/regions/catalog.js) 用全称 `"江苏省"`——`<option>` 找不到对应 value
2. `trace_node.city` 列存在 mojibake 字节（`"南京�\udc82"` 带替换符 + 孤立 UTF-16 低代理位），任何匹配尝试都会失败

**修复**：[ScanFlowDialog.vue](frontend/src/features/trace/components/ScanFlowDialog.vue) 的 `watch(selectedTaskId)` 加 `normalizeProvinceToRegionValue` + `normalizeCityToRegionValue` 两个本地归一化函数，先剥离 mojibake 字节 + 反查 REGIONS 标准 value，再写回 formData。修复纯前端，**不动 DB**——重跑 seed 脚本时不需要二次修补。

## 五、找替代测试码（如果 §2 的码不存在了）

### 5.1 SQL 直查（推荐）

打开 MySQL CLI：

```sql
-- 找：任意 IN_STOCK 码 + 对应开放的 OUTBOUND 任务
SELECT
  ts.trace_code,
  ts.current_node,
  ts.current_status,
  tft.id           AS task_id,
  tft.task_no,
  tft.status       AS task_status,
  tft.actual_quantity,
  tft.expected_quantity,
  src_n.node_name  AS source_node,
  tgt_n.node_name  AS target_node
FROM trace_snapshot ts
JOIN trace_node src_n
  ON src_n.node_name = ts.current_node OR src_n.node_code = ts.current_node
JOIN trace_flow_task tft
  ON tft.source_node_id = src_n.id
JOIN trace_node tgt_n ON tgt_n.id = tft.target_node_id
WHERE ts.current_status = 'IN_STOCK'
  AND tft.task_type = 'OUTBOUND'
  AND tft.status IN ('CREATED', 'PROCESSING')
  AND tft.actual_quantity < tft.expected_quantity
LIMIT 10;
```

任选一行就是一组可用 demo 组合。

### 8.2 提交时的实际 HTTP 调用

**选了任务**：

```
POST /api/trace-flow-tasks/9/scan
{
  "trace_code": "TC-EXT-0030-0005-450bde4a",
  "event_time": "2026-05-11T03:45:00",
  "remark": "运单驱动扫码 demo",
  "idempotency_key": "<crypto.randomUUID()>"
}
```

后端 [`TraceFlowTaskServiceImpl.scanTask`](backend/src/main/java/com/example/trace/service/impl/TraceFlowTaskServiceImpl.java) 处理：
1. ensureTaskOpen → status 不能是 COMPLETED/CANCELLED/EXCEPTION
2. resolveTaskScanPlan → 根据 task type + snapshot status 派生 ScanPlan
3. 自动派生 actionType=OUTBOUND, fromNode=源节点, toNode=目标节点, province/city 从节点取
4. 调 `traceScanRetryExecutor.executeAndReturnCreated` 走乐观锁重试 + REQUIRES_NEW 写链
5. 写 `trace_flow_task_scan` 明细 + 累加 `actual_quantity`

**没选任务**：

```
POST /api/traces/TC-EXT-0030-0005-450bde4a/events
{
  "action_type": "OUTBOUND",
  "event_time": "2026-05-11T03:45:00",
  "remark": "..."
  // 注意：fromNode/toNode/province/city 全部省略，让后端补齐
}
```

后端 [`TraceScanTransactionService.executeAndReturnCreated`](backend/src/main/java/com/example/trace/service/impl/support/TraceScanTransactionService.java) 处理：
1. authorizeAndResolveRoute：用 warehouse 默认绑定节点补齐 fromNode（如果没传），并从节点取 province/city
2. 写 `trace_lifecycle_log` + 更新 `trace_snapshot`
3. **不写** `trace_flow_task_scan`，运单进度不变

## 九、答辩讲稿建议

如果答辩问到这部分功能，建议这么讲：

> "项目的扫码工位（ScanHub）原本是孤立的——扫码后只能登记事件，不能关联到正在执行的发货单/接收单。但项目里其实有完整的「流转任务」模型（trace_flow_task + scanTask），有独立的任务工作台页面。我们把这两条链路在弹窗内接通了：
>
> 1. 扫码后弹窗自动调 `/candidate-flow-tasks` 查这个码当前可参与的开放运单；
> 2. 候选按 `compatibleActionType` 过滤——只展示与用户选定的扫码动作一致的；
> 3. 用户选定运单后，前端按运单的 source/target/province/city **自动填表**——这里有个细节，DB 里省份存的是简称、城市有 mojibake，前端做了 REGIONS 反查归一化；
> 4. 提交时**前端按用户是否选了运单做路径分流**：选了 → 走 `/trace-flow-tasks/{id}/scan` 联动 actualQuantity；没选 → 走 `/events` 原方案 A 路径，由后端 authorizeAndResolveRoute 用绑定节点补齐字段。
>
> 这种"叠加而非替换"的设计避免了同一份验证逻辑两套漂移——任务扫码的所有校验（容量、状态机、节点匹配）仍归 TraceFlowTaskServiceImpl 一处管理。"
