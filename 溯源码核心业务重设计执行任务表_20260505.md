## 二、设计共识

### 2.1 运输逻辑按“快递/物流发货”理解

厂家生产完成、准备运输时，通常已经知道货物要运到哪里。例如：

```text
北京工厂 → 上海仓库
```

因此系统不应让一线人员每次扫码后手填 `fromNode / toNode`。更合理的流程是：

```text
创建发货/流转任务
   ↓
任务中已经包含起点、终点、预计数量
   ↓
用户扫码
   ↓
系统根据当前用户、当前节点、当前任务、货物状态判断可执行动作
   ↓
自动填充节点、时间、位置
   ↓
用户一键确认
```

这与“快递单已知收件地址，揽收/中转/签收只做确认扫描”的方式一致。

### 2.2 批次赋码与一物一码不冲突

批次赋码的正确含义是：

> 一次性为 N 个实物生成 N 个唯一单品溯源码。

而不是：

> 一个生产批次共用一个溯源码。

推荐关系：

```text
赋码批次 ASSIGN-001
├── traceCode A → 实物 1
├── traceCode B → 实物 2
├── traceCode C → 实物 3
└── traceCode D → 实物 4
```

批次是管理单位，单品码是追溯单位。

### 2.3 用户体验目标

前线用户的理想操作是：

```text
打开任务 → 扫码 → 系统自动判断 → 一键确认
```

而不是：

```text
扫码 → 选择动作 → 手填起点 → 手填终点 → 手填地区 → 手填时间 → 提交
```

复杂输入只应出现在异常、纠错、临时流转等少数场景。

### B02 新增溯源码状态机策略 `TraceTransitionPolicy`

- 优先级：P0
- 目标：禁止任意状态随意追加任意扫码事件。

  - 新增 `TraceTransitionPolicy`
  - 改造 `TraceScanTransactionService`
  - 更新 `TraceStatus`
  - 更新 `ActionType`
- 完成标准：
  - 明确定义当前状态、事件、新状态三元组。
  - 非法流转返回明确业务错误。
  - `EXCEPTION`、`CORRECTION` 有独立规则。
- 验证方式：
  - 单元测试覆盖合法流转、非法流转、异常冻结、纠错不变更状态。

### B03 设计并实现“可执行动作”判断服务

- 优先级：P0
- 目标：支持扫码后系统判断“当前用户可以做什么”。
- 建议接口：
  - `GET /api/traces/{traceCode}/available-actions`
- 返回内容建议：
  ```json
  {
    "traceCode": "...",
    "currentStatus": "IN_TRANSIT",
    "currentNode": "北京工厂",
    "recommendedAction": "RECEIVE",
    "availableActions": [
      {
        "actionType": "RECEIVE",
        "label": "确认入库",
        "requiresRemark": false
      }
    ]
  }
  ```
- 完成标准：
  - 根据当前状态、用户权限、用户节点、是否存在流转任务返回动作。
  - 没有可执行动作时返回原因。
- 验证方式：
  - 单元测试覆盖生产人员、仓库人员、物流人员、普通查询用户。

### B04 改造扫码写入逻辑为后端自动推导节点

- 优先级：P0
- 目标：减少前端手填 `fromNode / toNode`。
- 重点范围：
  - `TraceScanTransactionService`
  - `ScanTraceRequest`
  - 后续节点/任务模型接入点
- 完成标准：
  - 常规流转中，`fromNode` 默认来自 `snapshot.currentNode`。
  - `toNode` 优先来自流转任务目标节点。
  - 前端传入节点只作为特殊场景，必须被后端校验。
- 验证方式：
  - 测试不传 `fromNode` 时可自动写入当前节点。
  - 测试伪造不一致节点时被拒绝或修正。

### B05 扫码接口增加幂等机制

- 优先级：P0
- 目标：防止二维码识别多次触发重复事件。
- 建议方案：
  - 请求增加 `idempotencyKey`。
  - 后端保存 `traceCode + actionType + idempotencyKey` 唯一记录。
  - 重复请求返回第一次处理结果。
- 完成标准：
  - 同一个扫码动作重复提交不会生成多条生命周期日志。
  - 并发重复请求也不会重复写入。
- 验证方式：
  - 并发测试或单元测试模拟重复提交。

## 九、阶段验收标准

### 阶段 1 验收

- 非法状态流转会被拒绝。
- 扫码重复提交不会产生重复日志。
- 常规扫码可以由后端推导节点。
- Hash/签名覆盖关键审计字段。
- 详情接口可以返回 effective/audit 两种视图。

### 阶段 2 验收

- 可以创建赋码批次。
- 一次批量生成 N 个唯一单品码。
- 未激活码不能流转。
- 标签打印、重打、作废有记录。
- 批次数量可对账。

### 阶段 3 验收

- 发货前已经知道目的地。
- 出库任务包含起点、终点、预计数量。
- 仓库/物流人员扫码时系统自动判断动作。
- 目标节点接收后形成闭环。

### 阶段 4 验收

- 一线用户常规操作不再手填 fromNode/toNode。
- 支持连续扫码。
- 可看到已扫/应扫数量。
- 无权限或不符合状态时提示明确原因。

### 阶段 5 验收

- 每个物品仍有唯一单品码。
- 支持箱码/托盘码聚合。
- 扫父码可批量流转子码。
- 查单品可看到聚合历史。
