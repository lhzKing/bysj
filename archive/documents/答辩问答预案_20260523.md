## 问题 1：你的系统用"哈希链 + RSA 数字签名"做防篡改，那它和真正的区块链有什么区别？为什么不直接上区块链？

**老师为什么会问**：溯源类毕设几乎一定会被对比到区块链，是否能讲清楚"链式哈希≠区块链"是检验你对密码学概念理解深度的关键。

**参考答案**：
两者的区别可以从三个层面说明。

1. **数据结构层面**：本项目和区块链相同的部分只有"链式哈希"——每条日志的 `current_hash = SHA256(本条业务数据 ‖ prev_hash)`，任何一条被篡改都会导致后续所有 hash 失配。这点和 Bitcoin 区块头里 `prev_block_hash` 的思想是一样的。
2. **共识层面**：区块链的核心其实不是"哈希链"，而是**去中心化共识**（PoW / PoS / PBFT）。本项目是单中心数据库写入，不存在多节点抢记账问题，所以没有挖矿、没有出块、没有最终一致性窗口。
3. **信任模型层面**：区块链假设"任意节点都可能作恶"，所以必须靠算力 / 权益 / 多数派来对抗。本项目假设"中心服务可信，但中心服务的管理员可能事后修改 DB"——所以我用**离线 RSA 私钥签名**让管理员也无法事后伪造：即使把 `trace_lifecycle_log` 整张表改了，公钥验签依然会失败（除非他同时拿到私钥，而私钥在生产 profile 下强制要求外置挂载，仓库里不存）。

**为什么没上链**：(1) 工业供应链场景中各节点（厂商、物流、仓储、终端）一般愿意接受一个**可信第三方平台**，不需要无主信任；(2) 链上吞吐和成本对扫码这种高频写入（每件配件 ~5 次扫描）不友好；(3) 学位论文工程量集中在密码学 + 并发 + 全栈三块，再叠 Fabric / 智能合约会失焦。

**留扩展口**：第三方验证接口 `GET /api/traces/public-key` 已经开放公钥，理论上把每条日志的 (hash, signature) 周期性锚定到一条公链（OP_RETURN）就能升级成"侧链"模式，这是论文"未来工作"里写的一条路径。

## 问题 2：既然有哈希链了，为什么还要 RSA 签名？只用 SHA-256 不行吗？

**老师为什么会问**：考察你是否真正理解 hash 和 signature 的区别——"防篡改"和"防伪造"是两个独立问题。

**参考答案**：
哈希链只能解决**完整性**问题（integrity），不能解决**身份认证**问题（authenticity）。

- 假设没有签名，只有 hash 链。攻击者如果拿到 DB 写权限，他可以把第 N 条到末尾**整段重写**——重新计算每一条的 `current_hash`，让链表面上完全自洽。验链时哈希全部对得上，但内容已经被篡改。
- 加上 RSA 签名后，每条日志的 `signature = SIGN(私钥, 业务数据)`。攻击者**没有私钥就无法生成合法签名**。即使他重算了 hash 链，逐条 `VERIFY(公钥, 数据, signature)` 时一定会失败。

换句话说：
- SHA-256 防"无意中被改坏"（如磁盘错误、误操作）；
- RSA 签名防"有意篡改"（如管理员作恶 / DBA 直连改库）。

二者**正交**，必须同时存在。在 [TraceLogFactory](backend/src/main/java/com/example/trace/service/impl/support/TraceLogFactory.java) 里可以看到这两步是先后顺序：先算 hash → 再签名。

## 问题 3：你的扫码流程用了乐观锁 + 外层重试 + REQUIRES_NEW，为什么不直接用 `SELECT ... FOR UPDATE` 行锁？这种"外层重试 + 内层 REQUIRES_NEW"的写法是不是过度设计？

**老师为什么会问**：这是项目里最有"技术含量"的一段，老师如果懂事务，一定会盯这个。如果你答不清，会被认为"项目是抄的"。

**参考答案**：
这个设计是**必须的**，不是过度设计。理由分两层：

**第一，为什么用乐观锁而不是悲观锁。**
- 业务侧：同一个 `trace_code` 在正常业务里几乎不会并发被两个节点同时扫——并发冲突是**偶发**而非常态，乐观锁的"无冲突零开销"更划算。
- 数据库侧：`FOR UPDATE` 会持有行锁直到事务结束，配合外部 RSA 签名 + 远程调用容易演变成长事务，拖垮 MySQL 连接池。乐观锁基于 `@Version` 字段，只在 UPDATE 那一瞬间比较版本号，几乎不持锁。
- 跨实例可移植：如果以后水平扩展到多实例，悲观锁需要分布式锁中间件（Redis / Zookeeper），乐观锁天然兼容多实例。

**第二，为什么"外层重试"必须在事务外、"实际写入"必须 `REQUIRES_NEW`。**
关键在 MySQL 默认的 **REPEATABLE READ** 隔离级别——同一事务内重复读相同行，看到的版本号是事务开始时的快照值，**不会因为别的事务 commit 而刷新**。所以：
- 如果把 while-retry 放在**同一个事务内**：第 1 次冲突回滚后，第 2 次读到的还是旧版本号 → 再次 UPDATE 还是 0 行 → 死循环。
- 正确做法是让重试发生在**两个独立物理事务之间**：旧事务先彻底回滚释放快照，下次进 `REQUIRES_NEW` 时开新事务、用新快照读到最新版本号。

代码上对应的就是：
```
scan()                          // 不标 @Transactional
  └─ for retry in 1..3:
       └─ doScanOnce()          // @Transactional(REQUIRES_NEW)
```
见 [TraceScanRetryExecutor](backend/src/main/java/com/example/trace/service/impl/support/) 和 [TraceScanTransactionService](backend/src/main/java/com/example/trace/service/impl/support/)。

**实测**：50 并发线程对同一 trace_code 扫码，重试上限 3 次的情况下成功率 100%，平均重试次数 1.4 次。

## 问题 4：你的 JWT 存在前端 `localStorage` 里，这不是有 XSS 风险吗？为什么不存 httpOnly Cookie？

**老师为什么会问**：这是 Web 安全的基础题。如果你回答"没有风险"，老师就知道你不懂；如果你能讲出**已知风险 + 补偿措施 + 迁移路径**，反而是加分项。

**参考答案**：
**承认风险**：是的，存 localStorage 的方案不抗 XSS——一旦页面被注入恶意脚本，攻击者 `localStorage.getItem('token')` 就能直接拿走 Token。理论上 httpOnly Cookie 是更安全的方案，因为 JS 读不到。

**为什么仍然选 localStorage**：这是个工程取舍。httpOnly Cookie 需要配套：
1. CSRF Token / SameSite 策略（Cookie 会自动随请求发送）；
2. 跨域时要处理 `withCredentials` 和 CORS 的 `Access-Control-Allow-Credentials`；
3. 手机扫码端是局域网 IP 直连，Cookie 的 `Secure`/`Domain` 配置会复杂化。

毕设阶段全部上 Cookie + CSRF 会让前后端鉴权链翻倍复杂，所以采取**补偿模式**抵御已知风险：

| 风险 | 补偿措施 | 代码位置 |
|---|---|---|
| Token 被窃 → 长期有效 | **短期 Token**（Access 2h / Refresh 1d） | `JwtUtil` |
| 用户改密码后旧 Token 仍可用 | `token_version` 字段，每次密码 / 角色变更递增，JWT 里嵌入版本号，请求时和 DB 对比 | `LoginInterceptor` |
| 服务端注销 / 强制下线 | **Redis 黑名单**，Token 一旦加入黑名单立即失效；Redis 故障时 **fail-closed** 直接返回 503，**不**降级放行 | `TokenStore` |
| XSS 注入风险面 | (1) `frontend/index.html` 设置 CSP meta；(2) 全站不使用 `v-html` / `innerHTML`（地图 InfoWindow 走 DOM API）；(3) 输入输出统一走 Vue 模板，享受自动转义 | `frontend/index.html` |

**迁移路径**：完整方案我已经写在 `docs/security/token-storage-and-csp.md`，下一阶段（生产化）会切换到 "httpOnly Refresh Cookie + 短期内存 Access Token" 的混合方案。

简而言之：**风险已知 → 影响面已估 → 补偿措施落地 → 迁移路径明确**，这是工程化决策，不是技术盲区。

## 问题 5：你为什么不用 Spring Security？自己写 JWT 拦截器是不是在重复造轮子？

**老师为什么会问**：技术选型题。Spring Security 是 Java Web 的事实标准，主动绕开它需要给出充分理由。

**参考答案**：
我评估过 Spring Security，结论是**对本项目过重**。

**Spring Security 适合的场景**：复杂的认证方式组合（Form / OAuth2 / SAML / LDAP）、需要细粒度过滤器链、需要 Method Security 高级表达式。

**本项目的实际需求**：
1. 单一认证方式：JWT Bearer Token，没有 Form 登录、没有 Session。
2. RBAC 是动态可配的——管理员在后台界面就能改"哪个角色拥有哪个权限"，需要从 DB 实时读，Spring Security 的 `@PreAuthorize("hasRole(...)")` 这种**编译期硬编码**的写法反而不适合。
3. 团队学习成本：Spring Security 的过滤器链有十几个，调试一个 401 经常要排查半小时；自定义 `LoginInterceptor` + `PermissionInterceptor` 只有两层、加起来 200 行，毕设答辩可控性更好。

**我的自定义方案具体做了什么**：
- [LoginInterceptor](backend/src/main/java/com/example/trace/security/LoginInterceptor.java)：解析 JWT → 校验签名 / 过期 → 校验 Redis 黑名单 → 比对 `token_version` → 注入用户上下文。
- [PermissionInterceptor](backend/src/main/java/com/example/trace/security/PermissionInterceptor.java) + `@RequirePermission` 注解 + `ApiPermissionMatcher`：扫描方法上的权限注解，从 DB 实时拉用户权限做匹配。`xxx:manage` 自动继承 `xxx:view`，由 `PermissionInheritanceResolver` 计算继承关系。

**结论**：不是"造轮子"，是"在通用框架不匹配业务的场景下，按需实现轻量替代"。这种取舍在工业实践中很常见，比如 Spring Cloud Gateway 也是自己写过滤器而非用 Spring Security。

## 问题 6：你的链验证接口 `GET /api/traces/{code}/verify` 是把这条溯源链整条遍历一遍——如果一个 SPU 有几千条扫码日志，性能怎么办？这是 O(n) 的，能上生产吗？

**老师为什么会问**：性能 / 复杂度题。考察你是否对自己代码的瓶颈有认知。

**参考答案**：
**先正面回答复杂度**：是 O(n)，单条链长度 n。验证过程要做三件事：(1) hash 链一致性比对；(2) 每条逐条 RSA 公钥验签；(3) 业务字段一致性比对。其中 RSA 验签是最耗时的环节，约 0.2~0.5ms / 条。

**这个 O(n) 能不能接受？**
- 单个 SPU 在工业场景下的扫码次数典型在 5~20 之间（生产→入库→出库→运输→签收→交付）。即使做到 50 条，总耗时 ~25ms，远低于一次扫码响应预算。
- 如果出现极端长链（比如 1000+），可以走以下优化路径：

**优化方案分级**：

1. **轻量缓存**（已设计未上）：对"历史已验证段"做摘要缓存。验链时把链拆成 `[0..k]`（已缓存）+ `[k+1..n]`（增量），只验后段。Redis Key：`trace:verify_checkpoint:{code}`，Value：`(last_idx, last_hash)`。
2. **Merkle 树**（论文未来工作）：把扫码日志组织成 Merkle 树而非线性链，验证某条记录只需 O(log n) 个兄弟节点。改造代价：日志表结构需要加 `merkle_path`，重建脚本要离线跑。
3. **分段签名**：每 100 条日志做一次"段头签名"（类似区块）。常规验证只验段头链，节省 99% 的 RSA 调用；要"精确验证"时再钻进段内部。

**结论**：当前 O(n) 是有意识的简化设计，业务场景下完全够用；优化路径论文里有阐述，是个**可扩展点**而非**遗留缺陷**。

## 问题 7：RSA 私钥这么关键，你的私钥放在哪？万一泄露怎么办？

**老师为什么会问**：密钥管理是密码学系统的"阿喀琉斯之踵"。99% 的密码学事故不是算法被破，而是密钥被偷。

**参考答案**：
密钥管理我分了**三个环境**，并且对 prod profile 做了**启动期自检**。

**配置策略**：
- `dev` / `test` profile：默认 `trace.signature.auto-generate=true`，启动时在 `D:/trace-runtime/keys/` 自动生成 2048 位 RSA 密钥对。代价是重启后旧签名失效——这是**显式接受的**，因为开发期不需要密钥稳定性。
- `prod` profile：默认值**全部清空**。`ProdProfileConfigGuard`（启动钩子）会强制检查：
  - 私钥路径不能为空；
  - 不能指向工作区路径（如 `backend/keys/*.pem`、`d:/bysj/...`）；
  - `auto-generate` 必须为 false；
  - 私钥文件必须能读取且 RSA 模长 ≥ 2048。
  任何一项不通过，**应用启动失败、不接客流量**。代码：[ProdProfileConfigGuard.java:34-115](backend/src/main/java/com/example/trace/config/ProdProfileConfigGuard.java)。

**仓库卫生**：早期 commit 历史里曾经把 `backend/keys/*.pem` 提交过，已经清理出工作区；`.gitignore` 已加 `*.pem`、`keys/`。

**泄露应急流程**（论文里列了）：
1. 立即生成新密钥对；
2. **保留旧公钥**（用于历史日志验证，历史是只读的）；
3. 新密钥从某个时间戳 T 开始用，T 之前用旧公钥，T 之后用新公钥；
4. 在 `trace_lifecycle_log` 加 `key_version` 字段，验签时按版本号选公钥。

**升级路径**：生产真正落地时应该接入 KMS（AWS KMS / 阿里云 KMS / 国密 HSM），私钥永远不出硬件——这是论文里"工程化部署"小节的内容。

## 问题 8：你这个系统跟现有的工业 ERP / MES 系统有什么区别？工业界为什么需要它？

**老师为什么会问**：实际应用价值题。如果讲不清楚业务意义，会被定义成"纯技术 demo"。

**参考答案**：
**核心区别在于"信任锚"的不同**。

- **ERP / MES**：解决"**企业内部**做什么、做了什么"的问题。数据 owner 就是企业自己，自己说自己的数据是真的，**内部信任**模式。
- **本系统**：解决"**跨企业**的链条上，谁该信谁的数据"的问题。一个配件从原厂出厂 → 经销商 A → 物流 → 装机厂 B → 终端售后，每一个环节都是**不同法人主体**，谁也不天然信任谁。本系统通过**哈希链 + 公钥签名**让每一个环节的数据都可以被**独立第三方验证**。

**典型业务场景**（论文里写的）：
1. **航空 / 汽车关键件召回**：发现批次缺陷时，反查这批件流向了哪些下游，哪些已装机、哪些还在库。传统做法靠人工对账，本系统秒级查询。
2. **质保索赔反欺诈**：客户拿一个配件来索赔，扫码立刻看出：这件是不是原厂的、有没有经过授权渠道、当前状态是否应该处于"已售出"。能识别窜货、翻新件、二次包装。
3. **跨企业供应链审计**：审计方拿公钥就能离线验证签名链，不需要任何一方"亮库"。

**和"区块链溯源"产品对比**：商业的蚂蚁链 / 京东链溯源核心也是 hash + 签名，只是把锚点放到链上。本系统的核心算法是同构的，只是信任模型选了"中心化 + 可验证"而非"去中心化共识"，工程复杂度低 1~2 个数量级，对中小企业更可落地。

## 问题 9：50 万件配件、每件平均 10 条日志，就是 500 万行 `trace_lifecycle_log`——这种数据量下，你的系统查询性能怎么保证？

**老师为什么会问**：考察你是否懂数据库索引、是否对项目的可扩展性边界有判断。

**参考答案**：
500 万行对 MySQL 来说**远未到瓶颈**，单表千万级以内只要索引设计合理，查询都是毫秒级。我从三个层次回答。

**第一，现有索引设计**（`backend/sql/init_schema.sql` 里可查）：
- `trace_lifecycle_log` 上 `(trace_code, scan_time)` 复合索引——溯源查询的高频维度是"按 code 查全链 + 按时间排序"，这个索引能命中绝大部分业务查询。
- `trace_snapshot` 上 `(trace_code)` 唯一索引——保证扫码时的快照定位是 O(log n)。
- 用户 / 角色相关表加 `username`、`role_code` 唯一索引。

**第二，已经做过的实测**：
- 后台有批量演示数据生成接口（`POST /api/admin/generate-sample-data?count=N`）。我跑过 50 万 SPU + 约 500 万日志的数据集（见 `数据库批量演示数据生成记录_20260511.md`），单条溯源链查询 < 30ms，列表分页查询 < 100ms。
- 链验证 50 条 ~25ms（前面问题 6 提过）。

**第三，到瓶颈之后的扩展路径**：
- **第一阶段（千万级）**：垂直拆分，把 `trace_lifecycle_log` 单独迁到一个 instance。
- **第二阶段（亿级）**：水平分表，按 `trace_code` 哈希分 32/64 表，配合 ShardingSphere。这里有个天然优势——溯源查询永远带 `trace_code` 作为查询条件，**没有跨片 JOIN**，分表代价低。
- **第三阶段**：冷热分离，半年前的日志归档到 ClickHouse / TiFlash，热数据继续留 MySQL。

**结论**：500 万行不是瓶颈，单表瓶颈大约在 2000~3000 万行（看硬件）；项目当前阶段不需要分库分表，但扩展路径清晰。

## 问题 10：你这个项目最大的创新点是什么？跟同类毕设比，独特在哪？

**老师为什么会问**：基本每场答辩都问。回答得不好会被定义为"组合现成技术做的 demo"。回答的关键是**不要吹**，而是**精准给出技术决策上别人没做或做错的点**。

**参考答案**：
我不会说"我发明了什么新算法"——本科毕设做不到，吹了反而扣分。我说的是**工程组合上的合理决策**，三个点：

**创新点一：在"中心化系统"里实现了"管理员也无法事后篡改"的密码学保证**。
大部分供应链溯源毕设要么纯 ERP 风格（没有防篡改），要么直接照搬 Hyperledger Fabric（运维门槛高）。本项目用**哈希链 + 离线 RSA 签名**这套轻量方案达到了"DBA 改库也无效"的效果，只要私钥安全，签名验证就是金标准。在不依赖外部信任锚的前提下，这是中心化方案能做到的最强保证。

**创新点二：扫码并发的"外层重试 + REQUIRES_NEW 内层事务"模式落地**。
这是 Spring 事务模型 + MySQL REPEATABLE READ + MyBatis-Plus 乐观锁三者交互的一个**正确做法**。看似简单，但很多教学项目（包括 GitHub 上不少高 star 的脚手架）都犯过"while-retry 套在同事务里"的错误——结果就是高并发下死锁或重试无效。我在测试用例里专门构造了 50 并发同 trace_code 的场景验证。

**创新点三：完整的"开箱即用 + 生产强校验"双模配置体系**。
很多毕设的运行门槛是"先按 readme 手工建库、生成证书、配密钥…"。本项目在 dev profile 下做到了"clone + 改 MySQL 密码 + `mvn spring-boot:run`"零阻碍启动；同时 `ProdProfileConfigGuard` 在 prod profile 下会启动期 fail-fast 拒绝弱配置（默认 JWT secret、root/root DB 凭据、工作区内的私钥路径、`auto-generate=true`）。**让开发体验和生产安全在同一份代码里共存**，是这个项目真正花了时间打磨的工程化细节。

**附**：完整代码、自动化测试、Postman 集合、CSP 决策文档、配置安全策略，仓库里都可查；不存在"PPT 实现"。

## 配套答辩 Tips

1. **先讲业务、再讲技术**：老师不一定都懂哈希链，开场 1 分钟讲清楚"为什么工业供应链需要这个"，能让后面的技术细节有锚点。
2. **代码可定位**：被问到任何技术点都能立刻报出文件路径——比如 RSA 签名是 [TraceLogFactory](backend/src/main/java/com/example/trace/service/impl/support/TraceLogFactory.java)、并发重试是 `TraceScanRetryExecutor`、Prod 校验是 `ProdProfileConfigGuard`。这能避免"你这是不是抄的"的怀疑。
3. **承认局限**：被问到"为什么不上区块链 / 为什么不用 httpOnly Cookie / 为什么验链是 O(n)"时，**先承认局限再讲补偿和扩展路径**。坦诚 + 有思考 > 强行辩护。
4. **不要碰自己没把握的领域**：如果老师问"你的国密 SM2/SM3 怎么集成"，可以直接答"当前用 RSA-2048 + SHA-256，国密替换是后续工作，主要工作是替换 `Signature.getInstance("SHA256withRSA")` 为 BouncyCastle 的 SM2 实现，但本项目还没做"。**不会就说不会**。
