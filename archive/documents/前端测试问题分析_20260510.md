## §1 扫码弹窗的来源节点/目标仓库全靠手填

> **2026-05-11 老师指正：** 表述需更精确——后端实际不是"完全没做运单"，而是做成了**两条独立的扫码链路**，普通弹窗那条没接运单上下文；后端节点绑定也不只校验、还有部分预填能力，但被前端必填校验挡死了。本节按指正重写。

### 1.1 现象

截图所示"入库登记"弹窗 6 个字段 5 个需要操作员手填：来源节点、目标仓库、省份、城市、入库时间、备注。出库 / 物流流转复用同一表单，只是 label 文案不同。

### 1.2 项目里实际有两条扫码链路

```
普通扫码链路（你看到的"扫码工位"）：
  ScanHub → /api/traces/{code}/available-actions
         → ScanFlowDialog（必填 fromNode/toNode/province/city）
         → POST /api/traces/{code}/events
  特点：要求前端填 fromNode / toNode / province / city

任务扫码链路（"仓库/物流任务工作台"，前端已有独立 view）：
  TraceFlowTaskWorkbench → POST /api/trace-flow-tasks/{id}/scan
  特点：任务 已知 sourceNode / targetNode，前端只传 traceCode + eventTime，后端自动构造 ScanTraceRequest
```

**也就是说：**
- 后端**不是**"运单完全没做"，而是任务扫码闭环已经做完
- 前端**不是**"完全没接"，TraceFlowTaskWorkbench.vue 已经接了任务接口
- **真正缺的**是：**普通扫码工位（ScanHub + ScanFlowDialog）尚未接入运单上下文**，扫码时不会自动匹配开放运单、不会用任务的 source/target 推导节点

### 1.3 代码定位（普通扫码链路）

### 1.5 后端节点绑定其实"既校验又预填"——只是被前端挡住了

老师指出我之前说"只校验不预填"不准。实际上 [`TraceUserNodeBindingServiceImpl.authorizeAndResolveRoute`](backend/src/main/java/com/example/trace/service/impl/TraceUserNodeBindingServiceImpl.java) 已经有以下补齐能力：

| 动作 | 缺什么 | 后端补齐策略 |
|---|---|---|
| INBOUND | 没传 toNode | 用用户**唯一/默认**绑定节点补齐 |
| OUTBOUND/TRANSFER | 没传 fromNode | 用用户**唯一/默认**绑定节点补齐 |
| 任意 | 没传 fromNode | 还会从 `trace_snapshot.current_node` 兜底补齐 |
| 任意 | 没传 province/city | 拿到操作节点后补齐 province/city |

**问题是** ScanFlowDialog 在前端把这 4 个字段强制必填：

```js
// ScanFlowDialog.vue 第 105-123 行
if (!formData.fromNode) { toast.error('请输入起始节点'); return }
if (!formData.toNode) { toast.error('请输入目标节点'); return }
if (!formData.province) { toast.error('请选择省份'); return }
if (!formData.city) { toast.error('请选择城市'); return }
```

后端的预填能力被前端校验挡死，永远走不到。**要解锁后端补齐，前端只需把这 4 个 if 拆掉**——让空字段直接发到后端，后端会自动补齐。

| # | 维度 | 数据源 | 控制点 |

**5 道门 AND 串联，缺一不可。**

### 2.5 为什么是 by design，不是 bug

实测三个业务账号的扫码权限分布：

| logistics | **trace:transfer** / trace:task:* / trace:exception:handle / trace:view | 物流流转 |

## §A.11 已知现状（by-design / 物理环境，不做代码改动）

### A.11.1 "打印"按钮无任何 UI 输出 — by-design 行业惯例 📝

[`api/trace.js:107`](frontend/src/features/trace/api/trace.js#L107) `printTraceCode` 仅 `POST /traces/{code}/print`，后端只在 `trace_lifecycle_log` 写一条 `PRINT_CODE` 事件 + 更新 `trace_code.print_count`，**不渲染标签 / 不生成 QR 图像 / 不调用浏览器打印**。

**为什么这么设计**：真实工业场景里"打印标签"由外接的 TSC / Zebra / 斑马热敏机器完成（USB 或网络）；系统的"打印"按钮职责是**审计记录**（操作员声明"我已贴了这张标签"），不是实际打印。这是行业惯例。

**如果想让演示效果更完整**（30-40 分钟工时）：
- `npm i qrcode`（约 10kb）
- 新建 `PrintLabelDialog.vue`：客户端生成 QR + 标签模板（追溯码 / 配件 / 批次 / 生产时间）+ `@media print` CSS + `window.print()` 触发
- "打印"按钮先弹 dialog → 用户确认后才调 `printTraceCode` API
- 评委可以用手机扫这个浏览器打印出的 PDF 上的 QR 直接进入溯源详情，全链路一气呵成

### A.11.2 笔记本摄像头扫不出"屏幕上"的 QR 码 — 物理环境问题 📝

[`QRScanner.vue:36-39`](frontend/src/shared/components/QRScanner.vue#L36-L39) `onDetect` 对 QR payload **零格式要求**，只要能解码出文本就 emit。所以扫不出**不是 payload 不兼容**，而是物理识别条件不足：

1. **屏幕反光 / 摩尔纹**：摄像头扫显示器上的 QR，刷新率 + 反光产生干扰条纹，识别率比扫纸张低 50%+
2. **QR 尺寸太小**：网上转的 QR 通常 200-300px，加屏幕缩放后超出大多数笔记本摄像头分辨率极限。建议屏幕上至少 400×400 像素，充满取景框 60%
3. **对焦距离**：笔记本前置摄像头多为定焦 ≈ 30cm，对屏幕小 QR 几乎对不上焦
4. **环境光不足 / 容错等级低**

**演示建议**：把 QR 截图发到**手机屏幕显示**（手机屏分辨率高、亮度高、不反光），笔记本摄像头扫手机屏；或者**打印到 A4 纸**（5cm 见方就够）。
