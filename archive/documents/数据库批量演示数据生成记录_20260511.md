## 二、生成策略（关键决策）

### 2.1 规模

### 2.2 哈希链 + RSA 数字签名
**关键约束**：项目用 SHA-256 哈希链 + RSA-SHA256 数字签名做不可篡改保证（`backend/src/main/java/com/example/trace/util/HashUtil.java`、`SignatureUtil.java`）。如果离线生成的签名钥匙跟后端运行时不一致，`GET /api/traces/{code}/verify` 会返回 `valid=false`。

**解决方式**：脚本直接读取 `D:/trace-runtime/keys/private_key.pem`（与后端 dev profile 配置 `TRACE_SIGNATURE_PRIVATE_KEY_PATH` 一致的那把密钥），用 Python `cryptography` 库实现完全一致的：

1. **Hash 算法**：复刻 `HashUtil.calculateHashInternal`，包括：
   - `LocalDateTime.truncatedTo(SECONDS)` 截断
   - `LocalDateTime.toString()` 行为（秒为 0 时省略 `:ss`）
   - `null` → `""`、`prev_hash=null/blank` → `"GENESIS"`、`operator=` 后缀、`remark=` 条件追加
2. **Signature payload**：复刻 `SignatureUtil.buildSignatureDataInternal` 的 `key=value|...` 拼接。
3. **RSA 签名**：PKCS1v15 + SHA-256，Base64 输出。

事前自检：拿了 DB 中已有一条 INIT log，Python 重算 hash 与 DB `current_hash` **完全相等**；用 Python 签名再用 Java 公钥验证 → 通过（自检脚本 [_self_check_hash_sign.py](scripts/_self_check_hash_sign.py)）。

事后端到端校验：随机抽 20 个 `TC-EXT-*` trace_code、共 125 条 log，每条都：
- 链式 `prev_hash` 与上一条 `current_hash` 一致 ✅
- Python 重算 hash = DB `current_hash` ✅
- RSA 公钥验签通过 ✅

校验脚本：[verify_chains.py](scripts/verify_chains.py)，输出：
```
[summary] traces=20 logs_total=125 hash_ok=125 signature_ok=125 chain_breaks=0
[PASS] all chains verified end-to-end
```

→ **意味着**后端 `/api/traces/{code}/verify`、`/api/traces/public-key` 对这批新数据全部正常返回 `valid=true`、`signatureVerifiedCount==totalLogs`。

1. **必须用本机 `D:/trace-runtime/keys/` 那把密钥的后端运行时**才能让这批数据通过 `/verify`。如果别的机器或临时切了 `auto-generate=true`，签名会因 keyId/keyVersion mismatch 失败 → 验签接口返回 `SIGNATURE_KEY_UNAVAILABLE`。

## 九、踩过的坑：第一次跑完后 `/verify` 仍然 `valid=false`（已修复）

### 9.1 现象

跑完 `seed_extended_data.py` 后，前端 Trace 详情页的"链上不完整 · 0/5 节点验签通过"，每个 log 都是 `SIGNATURE_INVALID`。但 Python 自检脚本（`verify_chains.py`）能通过 — 表现矛盾。

### 9.2 根因：后端运行时密钥 ≠ 磁盘密钥

诊断方法：调用 `GET /api/traces/public-key` 拿后端运行时公钥，再读 `D:/trace-runtime/keys/public_key.pem` 比对：

```
backend pubkey prefix: MIIBIjAN...rsq2x4M7rPJo15yA9ih6
disk    pubkey prefix: MIIBIjAN...0etR/XYGtMzGaziH
MATCH: False
```

→ 两把不是同一把。

`backend/src/main/resources/application-dev.yml` 原默认值：

```yaml
private-key-path: ${TRACE_SIGNATURE_PRIVATE_KEY_PATH:}    # 默认空
auto-generate:   ${TRACE_SIGNATURE_AUTO_GENERATE:true}    # 默认 true
```

启动后端时**没设这俩环境变量** + path 默认空 → `SignatureUtil.loadOrGenerateKeyPair` 走 `auto-generate` 分支，用一把每次启动**生成在内存里的临时密钥**签验。私钥永远导不出来 — 我们离线用磁盘密钥签的所有 3621 条 log，自然过不了这个后端。

Python 自检之所以"通过"，是因为它**也用磁盘密钥**做的 round-trip：磁盘私钥签 → 磁盘公钥验。两边密钥一致，但跟后端无关，自检结果是假阳性。

### 9.3 修复

修改 `backend/src/main/resources/application-dev.yml` 把默认值固化到磁盘密钥：

```yaml
private-key-path: ${TRACE_SIGNATURE_PRIVATE_KEY_PATH:D:/trace-runtime/keys/private_key.pem}
public-key-path:  ${TRACE_SIGNATURE_PUBLIC_KEY_PATH:D:/trace-runtime/keys/public_key.pem}
auto-generate:    ${TRACE_SIGNATURE_AUTO_GENERATE:false}
```

→ 重启后端后启动日志变成 `Loaded RSA key pair from external files`（之前是 `Generated temporary in-memory RSA key pair`），且 `keyId=default, keyVersion=1` 元数据匹配我们 log 行里写的一致。

跨平台：其他人的开发机如果没有 `D:/trace-runtime/keys/`，可以通过环境变量覆盖：
```bash
export TRACE_SIGNATURE_PRIVATE_KEY_PATH=
export TRACE_SIGNATURE_PUBLIC_KEY_PATH=
export TRACE_SIGNATURE_AUTO_GENERATE=true
```
（前两条清空 path、第三条恢复 in-memory 模式）

### 9.5 教训

> 自检脚本用同一把磁盘密钥做 sign + verify 是**自我背书**。要证明数据"对得上后端"，必须用**后端运行时实际的公钥**（通过 `/api/traces/public-key` 拿）来 verify，或者用后端 `/verify` 直接打。这次修复后 `verify_chains.py` 仍然只用磁盘公钥，但 application-dev.yml 已经把后端密钥固化到磁盘，二者归一，自检与端到端验证等价。
