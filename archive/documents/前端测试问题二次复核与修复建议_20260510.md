## 3. 发现 1（P1）：ScanFlowDialog 仍在做“纯手填”，但后端已能自动解析一半以上字段

### 3.1 代码证据

- `frontend/src/features/trace/components/ScanFlowDialog.vue:63-83`：`formData` 每次打开都清空 `fromNode/toNode/province/city`。
- `frontend/src/features/trace/components/ScanFlowDialog.vue:104-124`：前端强制要求 `fromNode/toNode/province/city/eventTime` 都填写。
- `frontend/src/features/trace/components/BaseFlowForm.vue:102-145`：来源/目标仍是自由输入，省市手选。

但后端已有自动解析能力：

- `backend/src/main/java/com/example/trace/service/impl/support/TraceScanTransactionService.java:224-235`：如果请求不传 `fromNode`，会用 `snapshot.currentNode` 自动补。
- `backend/src/main/java/com/example/trace/service/impl/TraceUserNodeBindingServiceImpl.java:160-188`：
  - `INBOUND`：可由用户唯一/默认绑定节点补 `toNode`。
  - `OUTBOUND/TRANSFER`：可由用户唯一/默认绑定节点补 `fromNode`。
- `backend/src/main/java/com/example/trace/service/impl/support/TraceScanTransactionService.java:277-283`：解析出操作节点后可自动补 `province/city`。
- `backend/src/main/java/com/example/trace/controller/UserController.java:61-67`：已有 `/api/users/me/trace-nodes` 可查询当前用户绑定节点。

### 3.2 这意味着什么

当前前端把“后端可推断字段”挡在浏览器表单前了。以 `warehouse` 扫 `d6f99...` 为例：

- 后端知道当前码在 `武汉东风汽车整装厂`（`snapshot.currentNode`）。
- 后端知道 `warehouse` 只绑定 `苏州中央仓储`。
- 对 `INBOUND` 来说，理论上可以提交最小 payload：

```json
{
  "actionType": "INBOUND",
  "eventTime": "2026-05-10T23:00:00",
  "idempotencyKey": "..."
}
```

后端会补出：

- `fromNode = 武汉东风汽车整装厂`
- `toNode = 苏州中央仓储`
- `province/city = 江苏 / 苏州市`

但现在前端要求操作员全部手填，所以用户体验仍像“手工登记单”，不像扫码工位。

### 3.3 最应该的修复推荐

**推荐先做 A 方案：用户绑定节点 + 当前快照节点预填。不要先做完整运单驱动。**

原因：当前 DB 中 `trace_flow_task = 0`，直接上运单匹配会先遇到“没有任务上下文”的演示空白；而节点绑定自动预填能立即消除 70% 手填字段，且后端已经支持。

#### A 方案实现步骤（建议优先）

1. 前端新增 API：`GET /api/users/me/trace-nodes`。
2. `ScanHub.vue` 在扫码得到 `available-actions` 后，同时加载当前用户绑定节点。
3. `ScanFlowDialog` 增加 props：
   - `currentNode`：来自 `actionDecision.currentNode`
   - `operationNodes`：来自 `/users/me/trace-nodes`
4. 打开弹窗时按动作预填：

| action | fromNode | toNode | province/city |
|---|---|---|---|
| `INBOUND` | 当前快照节点 | 用户默认/唯一绑定节点 | 绑定节点省市 |
| `OUTBOUND` | 当前快照节点/绑定节点 | 仍让用户选择目标节点 | 绑定节点省市 |
| `TRANSFER` | 当前快照节点/绑定物流节点 | 仍让用户选择目标节点 | 绑定节点省市 |

5. `BaseFlowForm` 从自由输入升级为“预填 + 可覆盖”：
   - from/to 显示为节点选择或 readonly chip。
   - 省市在能由节点确定时 readonly。
   - 保留高级覆盖入口，避免散扫场景卡死。
6. 调整前端校验：不要再无条件要求所有字段；至少允许后端能推断的字段为空。

#### 建议验收

- `warehouse` 扫 `d6f99...`：动作区出现“确认入库”；弹窗自动填 `from=武汉东风汽车整装厂`、`to=苏州中央仓储`、`省市=江苏/苏州市`。
- `superadmin` 未绑定节点：仍为空，但文案提示“请绑定节点或切换业务账号”。
- `producer` 扫该码：不要再预期 INBOUND；文案应提示当前角色无该动作权限或节点不匹配。

## 4. 发现 2（P1）：superadmin 空动作不是 bug，但当前 demo 数据会误导演示

### 4.1 代码和数据事实

- `TraceAvailableActionService.java:93-108`：状态机动作 → 角色权限 → 节点绑定三段过滤。
- `TraceAvailableActionService.java:212-214`：当角色有动作权限但节点过滤后为空，返回“当前用户未绑定当前可操作节点”。
- `TraceUserNodeBindingServiceImpl.java:142-158`：需要节点授权的动作，如果用户没有绑定节点，直接 `false`。
- 当前 DB：`superadmin/admin` 的绑定数都是 0。

### 4.2 推荐修复

我不建议在代码里让 SUPER_ADMIN 绕过节点绑定；那会破坏“功能权限”和“作业地点”两条线的模型。

**推荐组合：**

1. **文案修复**：`ScanHub.vue` 空状态根据当前用户/绑定状态展示更具体提示。
2. **demo 数据修复**：如果答辩默认用 `superadmin`，就在 demo seed 中给 `superadmin` 绑定全部节点。
3. **演示脚本修复**：明确：赋码/打印/激活用 `superadmin` 或 `producer`，仓库入库用 `warehouse`。

可选 demo SQL：

```sql
INSERT INTO trace_user_node_binding (user_id, node_id, org_id, default_node, enabled)
SELECT u.id, n.id, n.org_id,
       CASE WHEN n.node_code = 'NODE-FACTORY-BJ' THEN 1 ELSE 0 END,
       1
FROM sys_user u
JOIN trace_node n
WHERE u.username = 'superadmin'
ON DUPLICATE KEY UPDATE
  org_id = VALUES(org_id),
  enabled = 1,
  default_node = VALUES(default_node);
```

如果只想保持模型纯粹，不给超管绑节点，则至少在 README / 演示脚本中写清楚：**扫码登记用 `warehouse/logistics` 等业务账号，不用 `superadmin`。**

## 8. 发现 6（P0）：后端测试当前失败，必须先修测试契约

本次执行：

```bash
cd backend
mvn -q test
```

结果：`Tests run: 385, Failures: 2`。

失败文件：`backend/src/test/java/com/example/trace/config/WebMvcSecurityPathContractTest.java`

原因：

- 生产代码 `WebMvcConfig.java:36-41` 已把 `/api/public/**` 加入 `LoginInterceptor` 白名单。
- 生产代码 `WebMvcConfig.java:48-52` 已把 `/api/public/**` 加入 `PermissionInterceptor` 白名单。
- 测试仍在 `WebMvcSecurityPathContractTest.java:28-29`、`:52-53` 用 `containsExactly(...)` 断言旧列表。

推荐修复：

1. 登录拦截器期望改为：

```java
.containsExactly("/api/auth/login", "/api/auth/register", "/api/traces/public-key", "/api/public/**");
```

2. 权限拦截器期望改为：

```java
.containsExactly("/api/auth/**", "/api/traces/public-key", "/api/public/**");
```

3. 增加断言：

```java
assertThat(loginMapping.matches(request("GET", "/api/public/traces/TRACE-001"))).isFalse();
assertThat(permissionMapping.matches(request("GET", "/api/public/traces/TRACE-001"))).isFalse();
```

这是一个小修，但会影响 CI/交付可信度，应排在 P0。
