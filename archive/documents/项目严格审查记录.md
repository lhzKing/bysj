# 溯源码系统严格审查记录

## 一、总体结论

该项目已经具备较高完成度，属于功能闭环较完整的毕业设计系统，而不是简单的页面演示或 CRUD 拼接项目。系统已经覆盖了：

- 溯源码生成
- 扫码流转
- 溯源详情查询
- Hash 链校验
- RSA 数字签名
- JWT + Redis 登录认证
- RBAC 权限控制
- Dashboard 可视化
- 摄像头扫码能力

但如果按严格代码审查标准来看，当前项目的主要短板不是“有没有功能”，而是：

- 自动化测试不足
- 工程化和部署配置不足
- 前端性能优化不足
- 部分核心类职责偏重
- 某些模块仍偏原型式实现

一句话总结：

> 这是一个完成度较高、业务亮点清晰的毕业设计原型系统；强项在业务链路和系统完整性，短板在测试、工程化和可维护性。

---

## 二、代码层面总体审查

### 1. Backend 优点

#### 1.1 分层结构较规范
后端目录分为：

- controller
- service
- mapper
- entity
- dto
- security
- config
- common

说明项目具备基本的后端架构意识，职责分层明确，便于答辩展示和后续维护。

#### 1.2 核心业务链设计完整
后端围绕溯源主链路构建了完整流程：

- 赋码初始化
- 扫码流转
- 快照更新
- 生命周期日志记录
- 溯源详情查询
- 链路校验与签名验证

其中“快照表 + 生命周期日志表”的模型设计是系统亮点，兼顾了查询效率和历史可追溯性。

#### 1.3 防篡改设计有明显亮点
系统不是普通管理系统，而是加入了：

- `prevHash/currentHash` 链式设计
- `verifyChain` 完整性校验
- RSA 数字签名验证

这部分使系统具备“数据可信校验”的技术亮点，适合毕业设计答辩重点展示。

#### 1.4 并发控制有一定深度
扫码流转使用了：

- 乐观锁
- 重试机制
- `REQUIRES_NEW` 独立事务

说明项目不只是静态 CRUD，而是对并发一致性有一定考虑。

#### 1.5 权限与认证设计较完整
已实现：

- JWT
- Redis Token 黑名单
- Token version 控制
- 用户状态校验
- RBAC 权限控制

这套方案明显优于一般毕业设计中“只做登录不做权限”的实现。

---

### 2. Backend 主要问题

#### 2.1 缺少自动化测试
虽然执行 `mvn test` 可以通过，但实际情况是：

- 没有有效的测试源码
- 没有单元测试
- 没有集成测试
- 没有并发测试

因此严格来说，这不是“测试通过”，而是“没有测试所以通过”。

#### 2.2 配置写死，环境迁移能力弱
`application.yml` 中直接写入了：

- MySQL 地址、用户名、密码
- Redis 地址
- JWT secret
- RSA 密钥路径

这导致系统强依赖本机环境，不利于环境切换、部署迁移和配置管理。

#### 2.3 核心 Service 责任偏重
`TraceServiceImpl` 承担了大量职责：

- 参数校验
- 业务流程控制
- Hash 计算
- 签名生成
- 并发重试
- 状态推导
- 链路验证

目前还能维护，但继续扩展后容易演变为臃肿类。

#### 2.4 部分业务规则偏硬编码
例如默认节点名、动作对应权限、状态推导规则等都主要写死在代码中。短期可用，但后续变更时维护成本会上升。

#### 2.5 Redis 黑名单方案缺少故障策略说明
`TokenStore` 在 Redis 访问失败时倾向于放行，这更偏可用性优先，而不是安全性优先。作为毕业设计可以接受，但如果老师追问“Redis 挂了怎么办”，需要提前准备解释。

---

### 3. Frontend 优点

#### 3.1 目录结构优于普通学生项目
前端采用了：

- `core`
- `features`
- `shared`

而不是全部堆在 `views` 和 `components` 下，说明具备一定模块化设计意识。

#### 3.2 页面功能覆盖完整
前端已覆盖：

- 登录
- Dashboard
- 溯源列表
- 溯源详情
- 扫码中心
- 用户管理
- 角色管理
- 配件管理
- 摄像头测试

说明前端不是纯展示壳，而是基本接通了完整业务。

#### 3.3 权限控制与后端配合较好
前端通过路由 `meta.permissions` 和 store 中的权限判断，对页面访问和按钮展示做了基础控制，与后端 RBAC 对应良好。

#### 3.4 扫码场景贴合业务主题
项目中包含：

- `QRScanner.vue`
- `CameraTest.vue`
- HTTPS/证书相关文件
- 扫码说明文档

说明扫码不是摆设，而是明确考虑了浏览器摄像头能力和移动端演示。

---

### 4. Frontend 主要问题

#### 4.1 打包体积过大
前端构建过程中已出现 chunk 过大的告警：

- 主包超过 1 MB
- Dashboard 包超过 1.7 MB

说明图表、地图等资源较重，性能优化空间明显存在。

#### 4.2 页面层职责偏重
例如 Dashboard 页面承担了：

- 多接口并发请求
- ECharts 注册
- option 组装
- 数据适配
- 页面交互

这会影响可维护性和可测试性。

#### 4.3 请求层和 UI 层耦合偏高
`request.js` 中除了网络请求外，还耦合了：

- toast 提示
- 登录失效跳转
- 错误文案组装

优点是省事，缺点是扩展性和复用性较弱。

#### 4.4 前端测试体系缺失
目前没有看到：

- 单元测试
- 组件测试
- E2E 测试

同时项目采用 JS 而不是 TS，静态约束相对有限。

## 三、毕业设计答辩视角总结

### 1. 系统亮点

#### 1.1 不是普通信息管理系统，而是完整的溯源闭环系统
本系统围绕工业企业配件在生产、入库、出库、流转等环节的全过程进行设计，实现了从赋码、扫码流转、状态更新到链路校验的完整闭环。

#### 1.2 采用“快照表 + 生命周期日志表”双层模型
系统通过快照表保存当前状态，通过生命周期日志表保存完整历史，实现了“查询效率”和“过程可追溯性”的兼顾。

#### 1.3 引入 Hash 链与 RSA 数字签名提升可信度
每条日志都链接前一条记录的 Hash，同时附带数字签名，可从链路连续性和签名有效性两个维度检测篡改。

#### 1.4 基于 JWT + Redis 实现较完整认证机制
系统不只支持登录，还支持：

- Token 失效控制
- 退出登录黑名单机制
- 密码/角色变化后的 Token 版本校验

#### 1.5 基于 RBAC 实现角色权限管理
系统预置了超级管理员、管理员、生产人员、仓储人员、物流人员、普通用户等角色，支持按角色控制接口访问和页面访问。

#### 1.6 具备可视化与扫码交互能力
系统实现了 Dashboard 看板、地图、趋势图、拓扑图以及摄像头扫码能力，既支持后台管理，也支持业务场景展示。

---

### 2. 系统不足

#### 2.1 自动化测试不完善
系统当前功能完整，但自动化测试仍然薄弱，后续应加强后端核心业务测试、接口测试和前端关键页面测试。

#### 2.2 配置管理偏开发环境
数据库、Redis、JWT、密钥路径等配置主要面向本地开发，环境分离和配置外置化仍需完善。

#### 2.3 前端性能还有优化空间
前端图表和地图资源较重，构建包较大，后续需要继续优化代码拆分与资源加载策略。

#### 2.4 系统更偏原型验证，尚非企业级成品
目前系统更适合作为高完成度毕业设计成果或业务原型，而不是直接用于复杂生产环境的企业级系统。

---

### 3. 后续可优化方向

- 补充后端单元测试、集成测试、并发测试
- 增加前端组件测试和 E2E 测试
- 实现多环境配置和部署脚本
- 引入 Docker 或容器化部署
- 优化前端图表资源拆分与加载性能
- 进一步细化后端服务职责边界
- 增加日志监控、审计日志和异常告警能力

## 四、逐文件补充审查

### 1. `backend/src/main/java/com/example/trace/service/impl/TraceServiceImpl.java`

#### 优点
- 是整个系统最核心、最有技术含量的类。
- 同时覆盖赋码、扫码流转、链验证、签名验证等关键流程。
- 乐观锁重试 + 独立事务的思路是正确的。
- `verifyChain()` 的设计非常适合当作系统亮点展示。

#### 问题
- 类职责过重，已经承担过多业务责任。
- `produceAssign()` 和 `doScanOnce()` 都很长，继续增加逻辑会明显膨胀。
- 签名组装、Hash 组装、状态推导、日志落库其实都可以进一步拆分。
- 目前可读性依赖注释，一旦注释不清晰，维护成本会升高。

#### 建议
- 后续拆分为“命令处理”“状态推导”“链路验证”“签名能力”等更小服务。
- 为 `verifyChain()`、`scan()`、`produceAssign()` 分别补充测试。

---

### 2. `backend/src/main/java/com/example/trace/controller/TraceController.java`

#### 优点
- 接口语义清晰，REST 风格基本合理。
- 基础权限和细粒度权限都有体现。
- 对旧接口保留兼容路径，兼顾了前端迁移。

#### 问题
- Controller 中开始混入较细的业务权限判断逻辑，例如根据 `ActionType` 推导细粒度权限。
- 这种逻辑更适合下沉到 service 或统一授权层，否则 controller 会逐渐变重。
- 兼容旧接口虽然有价值，但会增加维护负担，后期需要有明确下线计划。

#### 建议
- 把 actionType -> permission 的映射独立出去。
- 在文档里明确旧接口的废弃时间或迁移策略。

---

### 3. `backend/src/main/java/com/example/trace/security/LoginInterceptor.java`

#### 优点
- 登录校验链条比较完整：Token 提取、黑名单、JWT 校验、用户状态、tokenVersion 校验、角色校验。
- 对未登录、用户禁用、Token 过期等情况都有较明确处理。

#### 问题
- 拦截器职责偏多，已经不仅是“登录拦截”，还承担了部分用户状态/角色有效性检查。
- `request.setAttribute(...)` 传递上下文虽常见，但类型安全较弱。
- 如果后续权限模型继续复杂化，这个类会进一步膨胀。

#### 建议
- 可逐步抽出“认证上下文对象”。
- 后续如果继续增强安全体系，可以考虑统一认证过滤链而不是继续堆拦截器逻辑。

---

### 4. `backend/src/main/java/com/example/trace/security/PermissionInterceptor.java`

#### 优点
- 支持方法级注解、类级注解、API 路径匹配三级校验。
- 权限判断思路清楚，满足毕业设计展示需求。

#### 问题
- `roleId == null` 时直接跳过校验，这种容错逻辑偏宽松。
- 依赖注解 + 路径权限双轨并行，后期容易出现权限来源分散的问题。

#### 建议
- 后续统一权限来源，减少“注解规则”和“路径规则”并存导致的理解成本。
- 对“跳过权限检查”的场景要更明确，避免将来误放行。

---

### 5. `backend/src/main/java/com/example/trace/security/TokenStore.java`

#### 优点
- 设计思路合理，使用 Redis TTL 处理黑名单很适合 JWT 场景。
- 功能完整，包括加入、校验、删除、清理、统计。

#### 问题
- Redis 异常时直接返回 false，意味着系统可能在依赖异常时放宽安全限制。
- `clearBlacklist()` 用 `keys`，不适合大规模场景，但在毕业设计里问题不大。

#### 建议
- 在论文或答辩中说明这是“面向原型环境的实现”。
- 如果要继续升级，可考虑监控 Redis 状态并定义更明确的失败策略。

---

### 6. `frontend/src/core/api/request.js`

#### 优点
- 统一封装请求、Token 注入和错误处理，减少页面重复代码。
- 对登录失败和 Token 失效做了区别处理，这点比较细。

#### 问题
- 请求层中耦合了 toast 和 router，网络层与展示层边界不够清晰。
- 某些页面若想自定义错误处理，会受到统一拦截器行为影响。

#### 建议
- 后续可将“默认提示行为”做成可配置策略。
- 将部分 UI 副作用从请求层剥离到更高层统一处理。

## 六、新增逐文件审查补充

### 12. `backend/src/main/java/com/example/trace/controller/AuthController.java`

#### 优点
- 登录、注册、退出、刷新 Token、获取当前用户、修改密码等链路完整。
- 登录流程中对用户状态、密码校验、权限列表返回都有考虑。
- 修改密码时同步提升 `tokenVersion` 并拉黑当前 Token，这个设计是合理的。

#### 问题
- Controller 中直接依赖多个 Mapper，说明部分认证业务尚未完全沉淀到 service 层。
- `AuthController` 已经在承担较多业务逻辑，而不只是请求分发。
- 注册逻辑较简单，缺少更细的约束，如密码强度、用户名规则扩展能力等。
- 刷新 Token 接口虽然可用，但在“旧 Token 何时刷新”“客户端如何协同”方面仍较偏实现层，而非完整会话策略。

#### 建议
- 后续可增加 `AuthService`，将认证领域逻辑从 Controller 中抽离。
- 补充认证流程测试，包括登录失败、Token 刷新、密码修改后旧 Token 失效等关键路径。

---

### 13. `backend/src/main/java/com/example/trace/service/impl/UserServiceImpl.java`

#### 优点
- 用户管理、角色优先级、特殊账号保护等规则比较完整。
- 能体现出项目对“管理员不能越权管理高权限用户”的约束意识。
- `tokenVersion` 与用户密码/角色变更联动，这点设计是加分项。

#### 问题

- `convertToResponse()` 存在潜在 N+1 查询问题：分页结果中每个用户再查一次角色。
- 用户管理规则虽然完整，但逻辑集中度较高，方法较长，后续继续扩展容易变重。
- 某些操作权限规则分散在多个方法里，后续统一性会变差。

#### 建议
- 后续可把“角色优先级/可操作性规则”抽成单独策略类。
- 查询列表时尽量一次性关联角色信息，减少额外数据库访问。

## 十三、整改优先级清单

下面将前四轮逐文件审查中发现的问题，按整改优先级分为 **P0 / P1 / P2**。

- **P0：必须优先处理** —— 会直接影响系统正确性、可维护性主干、答辩可信度
- **P1：建议尽快处理** —— 不一定马上出错，但会持续拖累后续修改和展示质量
- **P2：可后续优化** —— 偏工程提升、规范收敛、长期质量建设

---

### P0：必须优先处理

#### P0-1. 后端缺少自动化测试
**涉及模块：** 整个 backend，重点是 `TraceServiceImpl`、认证权限链路、链路校验逻辑

**问题表现：**
- `mvn test` 虽然通过，但基本没有测试源码
- 核心业务没有单元测试、接口测试、并发测试

**影响：**
- 当前系统稳定性主要依赖手工测试

#### PrimeVue UI 重构建议策略

建议采用 **“分层替换，而不是一次性全量推翻”**：

1. **先定基线**：主题、颜色、表单/按钮规范
2. **先改基础组件**：BaseButton / BaseInput / Card / Dialog / Toast
3. **再改布局层**：Login / MainLayout
4. **再改管理页**：User / Role / Part
5. **最后改业务页**：Trace / Scan / Dashboard

这样做的原因：
- 风险更可控
- 样式统一效果更明显
- 不会一开始就动到最复杂的业务交互页
