# 示例数据生成 · 完整实施计划（v3 · 20260527）

> **目标**：让 `POST /api/admin/*` 系列接口成为**唯一**的示例数据生成入口。schema_consolidated.sql 刚执行完的干净库，调完接口后立即可用，**不再依赖 Python 脚本**。
> **基准**：schema_consolidated.sql（15 张表）+ `scripts/seed_extended_data.py`（业务逻辑参考实现）。
> **原则**：生成的数据要"看着像真实生产"，不只是过 schema 约束。

## 二、接口设计

只动 admin 接口，**3 个**：

### 2.1 `POST /api/admin/seed-master-data`（新增）

种 4-7 号表的主数据（demo 用户 / 节点 / SPU / 绑定）。**幂等**：按业务键 (username / node_code / part_code / user-node 唯一对) 跳过已存在。

```
权限：trace:data:seed-master（新增）
返回：
{
  "code": 0,
  "data": {
    "demoUsers":          { "inserted": 4, "skipped": 0 },
    "traceNodes":         { "inserted": 18, "skipped": 0 },
    "partSpecs":          { "inserted": 15, "skipped": 0 },
    "userNodeBindings":   { "inserted": 16, "skipped": 0 }
  }
}
```

### 2.2 `POST /api/admin/generate-sample-data?count=600`（重写）

种 8-15 号表的业务数据。**前置检查**：trace_node / base_part_spec 不为空，否则 400 提示先调 seed-master-data。

执行顺序（一次调用全做）：
1. 生成 `count/20` 个 batch（按 SPU 分桶）
2. 生成 `count` 个 trace_code + lifecycle_log 链 + snapshot
3. 生成 `60` 个 flow_task + scan（基于已生成的 trace_code）
4. 生成 `~30` 组 aggregation（基于已生成的 trace_code）

```
权限：trace:data:generate（复用）
返回：
{
  "code": 0,
  "data": {
    "batches": 30,
    "traceCodes": 600,
    "lifecycleLogs": 3500,
    "snapshots": 600,
    "flowTasks": 60,
    "flowTaskScans": 700,
    "aggregations": 330,
    "durationMillis": 12000
  }
}
```

**flowTasks / aggregations 数量是固定的**（不随 count 线性增长），因为：
- 60 个任务足够展示 4 类型 × 5 状态的分布
- 30 个聚合组足够演示装箱码垛层级
- 业务逻辑上"每月开 60 个运单"是合理量

count 控制的就是单品码数量。

### 2.3 `DELETE /api/admin/clear-trace-data?confirm=DELETE_TRACE_DATA`（修复）

清 8-15 号表的全部业务数据，**保留**主数据（用户 / 节点 / SPU / 绑定）。

按 FK 顺序删：
```
trace_flow_task_scan → trace_flow_task → trace_aggregation
→ trace_scan_idempotency → trace_lifecycle_log → trace_snapshot
→ trace_code → trace_assign_batch
```

```
权限：trace:data:clear（复用）
返回：每张表的删除行数
```

## 三、典型使用流程

新装环境的同学从 0 开始：

```bash
# 第 1 步：起服务，schema_consolidated.sql 把库结构 + RBAC 灌好
mysql -u root -p < backend/sql/schema_consolidated.sql
cd backend && mvn spring-boot:run -Dspring-boot.run.profiles=dev

# 第 2 步：登录拿 token
TOKEN=$(curl -s -X POST http://localhost:8080/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"superadmin","password":"superadmin123456"}' | jq -r .data.token)

# 第 3 步：种主数据（节点 + SPU + demo 用户 + 绑定）
curl -X POST http://localhost:8080/api/admin/seed-master-data \
  -H "Authorization: Bearer $TOKEN"

# 第 4 步：灌业务数据（默认 600 单品，约 12 秒）
curl -X POST "http://localhost:8080/api/admin/generate-sample-data?count=600" \
  -H "Authorization: Bearer $TOKEN"

# 完成。打开前端登录 superadmin / producer / warehouse / logistics / user 都能看到完整数据
```

**整个过程不需要 Python**。如果以后想重灌：

```bash
curl -X DELETE "http://localhost:8080/api/admin/clear-trace-data?confirm=DELETE_TRACE_DATA" \
  -H "Authorization: Bearer $TOKEN"
# 主数据保留，可以直接再调 generate-sample-data
```

`scripts/seed_extended_data.py` 不删除，作为参考实现 + 离线兜底（万一服务起不来仍可用 Python 灌）。

## 四、业务逻辑约束清单（生成数据时必须遵守）

这部分是用户说"不能有能生成但逻辑错误"的具体落地。

### 4.1 哈希链 + 数字签名

- 每条 lifecycle log 的 `prev_hash` = 上一条同 trace_code log 的 `current_hash`，第一条 = `"GENESIS"`
- `current_hash` 由 `HashUtil.calculateHash(...)` 真算（**和 Python 脚本 / TraceLogFactory 完全一致**：B06 版含 operator，字段顺序固定）
- `signature` 由 `SignatureUtil.sign(buildSignatureData(...))` 用进程内注入的 RSA 私钥真签
- `signature_key_id` / `signature_key_version` 从 `SignatureUtil.getKeyId()` / `getKeyVersion()` 读
- 调 `GET /api/traces/{code}/verify` 必须返回 `valid=true`

### 4.2 trace_code ↔ trace_snapshot ↔ lifecycle 链 三者一致

| 字段 | 一致性约束 |
|---|---|
| `trace_code.code_status` | 映射自 snapshot.current_status：INIT→GENERATED, ACTIVATED→ACTIVATED, IN_STOCK→IN_STOCK, IN_TRANSIT→IN_TRANSIT, TRANSFERRED→IN_STOCK（已交付按在库处理）, EXCEPTION→EXCEPTION |
| `trace_code.activated_time` | = lifecycle log 中 ACTIVATE_CODE 那条的 `event_time` |
| `trace_code.activated_by` / `_username` | = ACTIVATE_CODE 那条的 operator（producer 用户）|
| `trace_code.print_count` | = lifecycle log 中 PRINT_CODE 出现次数（演示默认 1） |
| `trace_code.current_snapshot_id` | = `trace_code`（按 v11 backfill 行为）|
| `trace_snapshot.last_log_id` | = 该 trace 的**最后一条** lifecycle log 的 id |
| `trace_snapshot.last_hash` | = 最后一条 lifecycle log 的 `current_hash` |
| `trace_snapshot.current_node` | = 最后一条 log 的有意义节点（PRINT_CODE/ACTIVATE_CODE/EXCEPTION_OPEN 等"原地"动作的 from_node；INBOUND/TRANSFER/OUTBOUND 的 to_node）|
| `trace_snapshot.last_event_time` | = 最后一条 log 的 `event_time` |
| `trace_snapshot.version` | = 0（新建快照，version 从 0 起）|

### 4.3 lifecycle 链业务动作序列（每个 trace 都跑一遍）

**默认链**（5-7 条 log）：
```
INIT          @ factory（producer 操作，from_node=NULL, to_node=factory.name）
PRINT_CODE    @ factory（producer 操作，from_node=factory, to_node=NULL，remark="生产工作台打印标签"）
ACTIVATE_CODE @ factory（producer 操作，同上，remark="出厂前激活"）
OUTBOUND      @ factory→warehouse（producer 操作，province=factory.province，IN_TRANSIT 状态）
INBOUND       @ warehouse（warehouse 操作，province=warehouse.province，IN_STOCK 状态）

# 60% 概率追加：
TRANSFER      @ warehouse→logistics（logistics 操作）

# 上一步触发后再 55% 概率追加：
TRANSFER      @ logistics→customer（logistics 操作，TRANSFERRED 状态）

# 5% 概率全链路末尾追加：
EXCEPTION_OPEN @ 末节点（沿用 operator，remark="质检发现异常，待复核"，EXCEPTION 状态）
```

这套和 Python 脚本完全一致。

### 4.4 operator 角色映射

| ActionType | operator 取值 |
|---|---|
| INIT / PRINT_CODE / REPRINT_CODE / ACTIVATE_CODE / VOID_CODE / PACK / UNPACK / PALLETIZE / UNPALLETIZE | producer（随机选 producer 用户）|
| INBOUND / OUTBOUND | warehouse |
| TRANSFER | logistics |
| EXCEPTION / EXCEPTION_OPEN / EXCEPTION_CLOSE / CORRECTION | 沿用上一条 log 的 operator（或 warehouse）|

### 4.5 trace_assign_batch 业务约束

| 字段 | 取值规则 |
|---|---|
| `batch_no` | `ASSIGN-EXT-{batch_seq:04d}-{spu_code}`，全局唯一 |
| `production_order_no` | `PO-{YYYY}-{batch_seq:04d}` |
| `quantity_requested` | 15-25 随机（同桶 trace 数）|
| `quantity_generated/printed/activated` | 全部 = `quantity_requested`（因为生成时一次性灌 + 打 + 激活）|
| `manufacturer_node_id` | 选一个 FACTORY 节点 id |
| `status` | `GENERATED` |
| `operator_id/_username` | 一个 producer 用户 |
| **CHECK 约束** | `quantity_generated/printed/activated ≤ quantity_requested`（满足）|

### 4.6 trace_flow_task 业务约束

| 字段 | 取值规则 |
|---|---|
| `task_no` | `TASK-EXT-{seq:04d}-{type}`，唯一 |
| `task_type` 分布 | OUTBOUND 25% / INBOUND 25% / TRANSFER 25% / RECEIVE 25%（轮循） |
| `status` 分布 | COMPLETED 25 / PROCESSING 15 / CREATED 10 / EXCEPTION 5 / CANCELLED 5（共 60）|
| `source_node_id ≠ target_node_id` | **必须**（CHECK 约束）|
| 节点选择 | OUTBOUND: warehouse→logistics；INBOUND: logistics→warehouse；TRANSFER: warehouse→warehouse；RECEIVE: logistics→customer |
| `expected_quantity` | 8-30 随机 |
| `actual_quantity` | 按 status：CREATED=0；CANCELLED=0；PROCESSING=1~expected-1；COMPLETED=expected（或差异）；EXCEPTION=0~expected |
| `discrepancy_type` / `_quantity` / `_reason` / `_time` | COMPLETED 任务 30% 概率设差异；EXCEPTION 任务必设 SHORTAGE；其他 NONE |
| `complete_time` | 仅 COMPLETED 任务 set |
| `cancel_time` | 仅 CANCELLED 任务 set |
| `create_by` / `_username` | warehouse / logistics 用户 |

### 4.7 trace_flow_task_scan 业务约束

| 字段 | 取值规则 |
|---|---|
| `(task_id, trace_code, action_type)` | UNIQUE，每码每任务每动作一条 |
| `action_type` | 来自 task.task_type 映射：OUTBOUND→OUTBOUND；INBOUND→INBOUND；TRANSFER→TRANSFER；RECEIVE→**INBOUND**（CHECK 约束不允许 RECEIVE）|
| `counted` | 1 |
| scan 数量 | 等于该 task 的 actual_quantity；CREATED/CANCELLED 任务**不**写 scan |
| 候选 trace_code | 仅取 `code_status IN ('IN_STOCK','IN_TRANSIT')` 的 trace_code |
| `scan_time` | 任务 create_time 之后 30 分钟 +，每条递增 2 分钟 |
| `operator_user_id/_username` | = 任务的 create_by/_username |
| `idempotency_key` | `{task_id}-{trace_code}-{action}` |

### 4.8 trace_aggregation 业务约束

| 关系 | 规则 |
|---|---|
| CARTON 关系 | 24 个箱码，每个箱码下 10-15 个单品 |
| PALLET 关系 | 6 个托盘码，每个托盘下 3 个箱码 |
| 候选 trace_code | code_status ∈ ('IN_STOCK', 'IN_TRANSIT')，不重复使用 |
| `parent_code` 命名 | 箱码 `CTN-EXT-{seq:03d}-{uuid6}`；托盘码 `AGG-EXT-PALLET-{seq:03d}-{uuid6}` |
| `relation_type` | 单品→箱 = CARTON；箱→托盘 = PALLET |
| `active` | 演示默认 1（不模拟解绑场景，保持简洁）|
| `(parent_code, child_code, active_marker)` | UNIQUE，因为 active=1 且不重复绑定，自然满足 |
| `create_by/_username` | producer 用户 |

### 4.9 trace_user_node_binding 业务约束

| 用户 | 绑定节点 | default |
|---|---|---|
| producer | NODE-FACTORY-BJ, NODE-FACTORY-QD | BJ=1, QD=0 |
| producer2 | NODE-FACTORY-SZ, NODE-FACTORY-TJ, NODE-FACTORY-XA | SZ=1, 其他=0 |
| warehouse | NODE-WAREHOUSE-SZ, NODE-WAREHOUSE-GZ, NODE-WAREHOUSE-NJ | SZ=1, 其他=0 |
| warehouse2 | NODE-WAREHOUSE-CQ, NODE-WAREHOUSE-CS, NODE-WAREHOUSE-SY | CQ=1, 其他=0 |
| logistics | NODE-LOGISTICS-SH, NODE-LOGISTICS-CD | SH=1, CD=0 |
| logistics2 | NODE-LOGISTICS-JN, NODE-LOGISTICS-FZ, NODE-LOGISTICS-ZZ | JN=1, 其他=0 |

共 **16 条**绑定。

## 七、关键技术点（实现细节）

### 7.1 两阶段提交（沿用 T-P1-01）

```
Stage 1（无事务，纯内存构建）：
  - 读 master data（node / part / demo user 列表）
  - 构造所有 batch / trace_code / log（含 hash + RSA 签名）/ snapshot
  - 构造所有 flow_task / scan
  - 构造所有 aggregation
  注意：trace_code.batch_id 此时还是 None；batch.id 由 DB 回填后再赋

Stage 2（分块 REQUIRES_NEW）：
  Chunk A: commit batches（30 行单次提交 OK）
  In-memory: 用 batch.id 回填每个 trace_code.batchId
  Chunk B..N: commit trace 单元（log + snapshot + trace_code 一起，每 chunk 50 trace）
  Chunk M..: commit flow_tasks（每 chunk 30 task）
  In-memory: 用 task.id 回填 scan.taskId
  Chunk M+1..: commit scans
  Chunk Final: commit aggregations
```

每个 chunk 失败不回滚已 commit 的 chunk（best-effort，与 T-P1-01 语义一致）。

### 7.2 哈希链构造（DemoChainBuilder 核心方法）

```java
public List<TraceLifecycleLog> buildChain(
        String traceCode, Long spuId,
        TraceNode factory, TraceNode warehouse, TraceNode logisticsNode, TraceNode customer,
        String producerOp, String warehouseOp, String logisticsOp,
        LocalDateTime initTime, Random rng) {
    List<TraceLifecycleLog> logs = new ArrayList<>(7);
    String prevHash = "GENESIS";
    LocalDateTime t = initTime;

    // INIT @ factory
    logs.add(append(prevHash = build(traceCode, spuId, INIT, null, factory.name, factory.province, factory.city, null, t, prevHash, producerOp).getCurrentHash()));
    // PRINT_CODE @ factory
    t = t.plusMinutes(10 + rng.nextInt(31));
    logs.add(build(traceCode, spuId, PRINT_CODE, factory.name, null, factory.province, factory.city, "生产工作台打印标签", t, prevHash, producerOp));
    prevHash = lastHash(logs);
    // ACTIVATE_CODE @ factory
    t = t.plusMinutes(5 + rng.nextInt(21));
    logs.add(build(traceCode, spuId, ACTIVATE_CODE, factory.name, null, factory.province, factory.city, "出厂前激活", t, prevHash, producerOp));
    prevHash = lastHash(logs);
    // OUTBOUND
    t = t.plusHours(2 + rng.nextInt(13));
    logs.add(build(traceCode, spuId, OUTBOUND, factory.name, warehouse.name, factory.province, factory.city, "发往" + warehouse.name, t, prevHash, producerOp));
    prevHash = lastHash(logs);
    // INBOUND
    t = t.plusHours(6 + rng.nextInt(37));
    logs.add(build(traceCode, spuId, INBOUND, factory.name, warehouse.name, warehouse.province, warehouse.city, "到仓签收", t, prevHash, warehouseOp));
    prevHash = lastHash(logs);
    // 60% TRANSFER
    if (rng.nextDouble() < 0.6) { ... }
    // 5% EXCEPTION_OPEN
    if (rng.nextDouble() < 0.05) { ... }
    return logs;
}
```

每个 build 内部调 `TraceLogFactory.createLog(...)`，由它做 hash + sign。**与 Python 脚本字段顺序完全一致**，因为都委托给 HashUtil / SignatureUtil。

### 7.3 trace_aggregation 字段处理

实体 `TraceAggregation` 有 `active`（写）和 `active_marker`（STORED GENERATED，**不写**）。需要在 entity 上加 `@TableField(exist = false)` 或类似机制；**先确认 MyBatis-Plus 默认是否会尝试写 active_marker，如果会就加 `@TableField(insertStrategy = FieldStrategy.NEVER)`**。

### 7.4 trace_snapshot 没有 create_time 列

entity 已正确（无 createTime 字段）。

### 7.5 prod profile 守护

- `TraceDemoDataProperties.enabled` 控制；prod 默认 false
- 三个 admin 接口都通过 `ensureAdminOperationEnabled` 检查

## 十、风险与备选

| 风险 | 缓解 |
|---|---|
| count=600 时 lifecycle log ≈ 3500，签名耗时大 | Python 脚本实测同规模 ~10 秒，Java 性能相近；不在事务内签名（stage 1 无事务）→ 不会持有连接 |
| Java BCryptPasswordEncoder 与 init_schema.sql 现有 4 个 BCrypt 哈希明文不一致 | seedMasterData 现场用 `<username>123456` 编码；首次种入即用新密码；如果以后想对齐旧哈希，把 Python 那 4 个常量字符串直接硬编码到 Java 也行 |
| trace_aggregation.active_marker 是 STORED GENERATED，MyBatis-Plus insert 时会试图带值 | 实施前在 TraceAggregation entity 的 activeMarker 字段（如有）上加 `@TableField(exist = false)` 或 `@TableField(insertStrategy = FieldStrategy.NEVER)`，避免 SQL 报错 |
| `trace_node.org_id` 是空字段，但代码中 `TraceUserNodeBinding.org_id` 要从 node 拷贝 | demo 阶段 `org_id` 永远 NULL，照拷即可 |
| 现有库已有数据时调 generate-sample-data 会和现有 trace_code 主键冲突 | trace_code 用 `TC-EXT-{batch:04d}-{seq:04d}-{uuid8}` 前缀+UUID 后缀，撞键概率约 0；retry 一次即可。**或者**：要求先调 clearTraceData |
