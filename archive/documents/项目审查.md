## 1. 总体结论

本项目并不是“架构完全混乱、必须推倒重写”的项目。后端已经形成了较清晰的 Controller / Service / Mapper / DTO / Entity 分层，溯源核心链路也拆分出了赋码、扫码事务、重试、验链、签名等独立组件；前端也有比较明确的 API 模块、请求统一转换层、Pinia 用户态和组件化布局。

但是，当前仓库状态还不适合作为可靠发布版本。主要风险集中在四类：

1. **安全与环境隔离风险较高**：默认 JWT 密钥、默认数据库凭据、DEBUG 安全日志、MyBatis stdout SQL 日志、固定本地 RSA 密钥路径，以及工作区内真实私钥同时存在。
2. **认证与权限边界存在漂移**：前端登录页注册调用的是后台用户管理接口 `/api/users`，而后端和文档定义的自助注册接口是 `/api/auth/register`；扫码权限模型的注释、实现、前端入口和权限继承也不完全一致。
3. **数据完整性与审计正确性存在明确缺陷**：配件删除没有检查溯源引用，schema 也没有 SPU 引用外键；事件 `remark` 前端发送但后端丢弃；时间解析失败时静默改为当前时间。
4. **测试与文档已经发生漂移**：后端单元测试当前通过，但前端测试当前红灯；README、backend README、api-doc、postman-guide 与实际实现存在多处不一致。

实测状态：

- 后端：在 `backend/` 执行 `mvn test -q`，当前通过。
- 前端：在 `frontend/` 执行 `npm run test:run`，当前失败：**20 个测试文件中 3 个失败，51 个测试中 5 个失败**。失败集中在 `useConfirm` 和 `MainLayout` 相关测试，原审查记录中还提示 `Login.test` / router-request-store 耦合需继续复核。

最终判断：

> **不建议整仓立即重写；建议立即做一轮以“配置安全、认证注册、Token 失效、扫码权限、事件契约、数据完整性、前端测试恢复”为核心的限定范围重构。**

## 2. 做得好的地方

### 2.1 后端整体分层和入口边界较清晰

- **位置**：`backend/src/main/java/com/example/trace/controller/TraceController.java:32-45`，`backend/src/main/java/com/example/trace/service/impl/TraceServiceImpl.java:19-40`

  - `TraceServiceImpl` 没有把赋码、扫码、验链全部揉成巨型方法，而是委托给 support 服务。
- **正面影响**：

  - 比所有业务都塞进 Controller 或单个 Service 更容易测试和演进。

## 3. 严重问题

### [严重] 默认密钥、默认凭据、固定私钥路径和实际私钥文件同时存在

- **位置**：`backend/src/main/resources/application.yml:6-8`，`backend/src/main/resources/application.yml:29`，`backend/src/main/resources/application.yml:34`，`backend/src/main/resources/application.yml:53`，`backend/src/main/resources/application.yml:59-63`，`backend/.env.example:22`，`backend/.env.example:32-37`，`backend/keys/private_key.pem:1`
- **现象**：
  - 数据库默认连接为 `root/root`。
  - JWT 默认密钥为固定值 `TraceSystemDefaultSecretKey2024!@#$`。
  - MyBatis 默认使用 stdout SQL 日志。
  - 安全包默认 DEBUG 日志。
  - RSA 私钥和公钥默认指向 `d:/bysj/backend/keys/...`。
  - 当前工作区实际存在 `backend/keys/private_key.pem`，且文件头为 `-----BEGIN PRIVATE KEY-----`。
- **影响**：
  - 如果默认 JWT secret 用于长期环境，Token 可被伪造。
  - 如果当前私钥用于真实签名环境，溯源签名的可信度会直接失效。
  - SQL stdout 与 DEBUG 安全日志可能泄露认证信息、SQL、参数和业务数据。
  - 固定本地路径降低环境可移植性，也容易把开发路径带入部署环境。
- **原因分析**：
  - 开发默认值、演示默认值、长期运行默认值没有通过 profile 严格隔离。
  - 缺少启动时 fail-fast 校验。
  - 当前目录不是 Git 仓库，无法确认私钥是否已进入版本历史，因此“是否已提交”必须待验证。
- **修复建议**：
  - 拆分 `application-dev.yml`、`application-test.yml`、`application-prod.yml`。
  - 生产 profile 禁止默认 JWT secret；未显式配置直接启动失败。
  - 生产 profile 禁止安全日志 DEBUG、禁止 SQL stdout。
  - 从工作区移除实际私钥，改由部署时挂载或密钥管理系统注入。
  - 轮换当前 RSA 签名密钥和 JWT secret。
  - `.env.example` 只保留占位符，不放可直接使用的真实密钥值。
- **是否建议立即处理**：是，P0。

### [严重] 登录拦截器在 DEBUG 下记录完整 Authorization Header

- **位置**：`backend/src/main/java/com/example/trace/security/LoginInterceptor.java:51-52`，`backend/src/main/resources/application.yml:51-53`
- **现象**：
  - `LoginInterceptor` 直接输出 `Authorization header: [{}]`。
  - 默认配置中 `com.example.trace.security` 日志级别为 DEBUG。
- **影响**：
  - 日志会包含完整 Bearer Token。
  - 日志一旦被控制台历史、错误报告、CI、运维平台或普通用户拿到，Token 可在有效期内被重放。
- **原因分析**：
  - 调试日志未做敏感信息脱敏。
  - 安全包默认 DEBUG 放大了该问题。
- **修复建议**：
  - 删除该日志，或仅记录 Header 是否存在、JTI / hash 摘要、Token 前后极少数字符。
  - 生产环境默认 `INFO` 或 `WARN`。
  - 增加静态扫描或测试规则，禁止日志打印 `Authorization`、`password`、`token` 原文。
- **是否建议立即处理**：是，P0。

### [严重] Token 黑名单 Redis 异常时 fail-open

- **位置**：`backend/src/main/java/com/example/trace/security/TokenStore.java:43-68`，`backend/src/main/java/com/example/trace/security/TokenStore.java:77-94`，`backend/src/main/java/com/example/trace/security/LoginInterceptor.java:61-65`，`backend/src/main/java/com/example/trace/controller/AuthController.java:171-174`，`backend/src/main/java/com/example/trace/controller/AuthController.java:214-215`，`backend/src/main/java/com/example/trace/controller/AuthController.java:305-310`
- **现象**：
  - `addToBlacklist` 捕获所有异常后只记录日志，不抛出。
  - `isBlacklisted` 捕获所有异常后返回 `false`。
  - 登录拦截器、登出、刷新旧 Token 作废、改密当前 Token 作废均依赖 TokenStore。
- **影响**：
  - Redis 不可用时，已登出的 Token 仍可继续使用。
  - 刷新后旧 Token 可能继续有效。
  - “登出后 Token 立即失效”的文档语义不成立。
- **原因分析**：
  - 黑名单是安全控制的一部分，但实现上把 Redis 异常当成普通缓存失败处理。
- **修复建议**：
  - 生产环境中 `isBlacklisted` 对 Redis 异常应 fail-closed，或明确返回“认证系统不可用”。
  - `addToBlacklist` 失败时，登出 / 刷新 / 改密链路应向调用方暴露失败。
  - 或改为数据库 `tokenVersion/sessionVersion` 统一承担强失效语义。
  - 增加 Redis 异常场景测试。
- **是否建议立即处理**：是，P0。

### [严重] 地图 InfoWindow 使用未转义业务字段拼接 HTML，存在存储型 XSS 风险

- **位置**：`backend/src/main/java/com/example/trace/service/impl/support/TraceScanTransactionService.java:57-69`，`frontend/src/features/trace/components/TraceRouteMap.vue:91-95`，`frontend/src/features/trace/components/TraceRouteMap.vue:228-239`
- **现象**：
  - 后端扫码事件会把 `fromNode`、`toNode` 等请求字段写入日志。
  - 前端地图点名称来自 `log.toNode || log.currentNode || ...`。
  - InfoWindow 通过模板字符串拼接 HTML，并直接插入 `${point.name}`。
- **影响**：
  - 如果节点名称包含 HTML / script payload，会被存入溯源日志。
  - 其他用户打开溯源详情地图时可能触发存储型 XSS。
  - 因前端 Token 存在 localStorage，XSS 后果会被进一步放大。
- **原因分析**：
  - 地图库 InfoWindow 接受 HTML 字符串，而业务字段进入 HTML 前未转义。
  - 后端对节点名称缺少长度、字符集、危险字符校验。
- **修复建议**：
  - 前端改用 DOM API 创建 InfoWindow 内容，并通过 `textContent` 设置业务字段。
  - 或对所有插入 HTML 的字段做严格 HTML escape。
  - 后端对 `fromNode`、`toNode`、`province`、`city` 增加长度和字符集校验。
  - 增加 XSS 回归测试。
  - 配合 CSP 限制脚本执行来源。
- **是否建议立即处理**：是，P0。

### [严重] 管理端清空溯源数据接口缺少环境隔离和二次保护

- **位置**：`backend/src/main/java/com/example/trace/controller/AdminController.java:24-33`，`backend/src/main/java/com/example/trace/service/impl/TraceDemoDataServiceImpl.java:98-112`
- **现象**：
  - `/api/admin/clear-trace-data` 仅用 `@RequirePermission("role:manage")` 保护。
  - 实现会删除全部溯源日志和全部快照。
  - `/api/admin/generate-sample-data` 的 `count` 没有明显上限。
- **影响**：
  - 具备角色管理权限的用户可以清空全部溯源业务数据。
  - 如果该接口暴露在长期运行环境，误操作或权限配置错误的破坏性很大。
  - 大量生成示例数据可能造成长事务、数据库膨胀或接口阻塞。
- **原因分析**：
  - 演示 / 运维接口与正常业务 API 放在同一运行面。
  - `role:manage` 与“清空业务数据”的高危语义不匹配。
- **修复建议**：
  - 仅在 dev / test profile 注册该 Controller，或拆成独立维护工具。
  - 增加专用高危权限，例如 `admin:dangerous` / `trace:data:clear`。
  - 增加二次确认参数、审计日志和操作告警。
  - 给 `generate-sample-data` 增加上限。
- **是否建议立即处理**：是，P0/P1；若系统会长期运行，按 P0 处理。

## 4. 主要设计缺陷

### [主要] 登录页自助注册调用后台用户管理接口 `/api/users`

- **位置**：`frontend/src/shared/components/Login.vue:7`，`frontend/src/shared/components/Login.vue:61-73`，`frontend/src/features/user/api/users.js:28-30`，`backend/src/main/java/com/example/trace/controller/AuthController.java:114-160`，`backend/src/main/java/com/example/trace/config/WebMvcConfig.java:33-40`，`README.md:341`，`api-doc.md:197-199`，`postman-guide.md:59-63`
- **现象**：
  - 登录页导入的是用户管理 API `createUser`。
  - 注册时调用 `/api/users`，并硬编码 `roleId: 2`、`status: 1`。
  - 后端已有公开注册接口 `/api/auth/register`。
  - 登录拦截器只放行 `/api/auth/login`、`/api/auth/register`、`/api/traces/public-key`。
  - README / API 文档 / Postman 文档均把用户注册描述为 `/api/auth/register`。
- **影响**：
  - 未登录用户点击注册大概率会得到 401。
  - 如果未来误放行 `/api/users`，会把自助注册和后台用户管理耦合在一起，权限边界脆弱。
  - 前端、后端、文档三者不一致，维护时容易反复出错。
- **原因分析**：
  - 自助注册和后台创建用户没有在前端 API 层隔离。
  - 默认角色使用前端硬编码 ID，而不是后端按 `role_code=USER` 统一分配。
- **修复建议**：
  - 新增前端 `register` API：`request.post('/auth/register', { username, password })`。
  - 登录页注册只调用 `/auth/register`，不传 `roleId/status`。
  - 后台用户创建继续使用 `/users`，仅管理员可用。
  - 删除前端硬编码 `roleId: 2`。
  - 增加 Login 注册流程测试和 AuthController 注册契约测试。
- **是否建议立即处理**：是，P1；如果自助注册是正式功能，则 P0。

### [主要] 配件删除没有业务引用检查，schema 也没有 SPU 外键兜底

- **位置**：`backend/src/main/java/com/example/trace/service/impl/PartServiceImpl.java:148-159`，`backend/src/main/java/com/example/trace/service/impl/PartServiceImpl.java:162-173`，`backend/sql/init_schema.sql:68-86`，`backend/sql/init_schema.sql:88-123`
- **现象**：
  - `deletePart` 删除前只检查配件是否存在。
  - 代码中明确留下 TODO，承认尚未检查是否被溯源记录引用。
  - `trace_snapshot.spu_id` 和 `trace_lifecycle_log.spu_id` 没有外键约束到 `base_part_spec.id`。
- **影响**：
  - 已参与溯源的配件规格可能被删除，历史记录语义悬空。
  - 溯源详情、统计、验链、报表可能出现孤儿数据。
- **原因分析**：
  - 业务完整性依赖应用层自觉，但应用层 TODO 未实现。
  - 数据库模型没有表达 SPU 与溯源事实之间的约束。
- **修复建议**：
  - 删除前查询 `trace_snapshot` 和 `trace_lifecycle_log` 是否存在 `spu_id=id`。
  - 有引用时拒绝删除，并返回明确业务错误。

## 7. 建议的修复优先级（P0 / P1 / P2）

### P0：立即修复，高风险安全或数据破坏问题

1. 移除工作区私钥，轮换 RSA 签名密钥。
2. 轮换 JWT secret，生产环境禁止默认密钥启动。
3. 删除或脱敏 Authorization Header 日志，安全日志默认非 DEBUG。
4. TokenStore Redis 异常从 fail-open 改为 fail-closed，或改用强一致 tokenVersion/sessionVersion 失效策略。
5. 修复 `TraceRouteMap.vue` InfoWindow HTML 注入风险。
6. 将 `/api/admin/clear-trace-data` 限定在 dev/test profile，或增加高危权限、二次确认和审计。
7. 关闭生产默认 MyBatis stdout SQL 日志，按 profile 管理 SQL 输出。

### P1：核心业务契约、权限和完整性问题

1. 重建注册链路：前端登录页注册改走 `/auth/register`，删除公共注册中的 `roleId/status`。
2. 统一扫码权限模型：Controller、权限继承、初始化 SQL、迁移脚本、前端按钮、README/API 文档同步收敛。
3. 统一事件契约：`remark` 要么真正落库并纳入签名，要么删除前端字段；`eventTime` 严格校验，不再 fallback 到 now。
4. 为配件删除补引用完整性保护，评估软删除、索引和外键策略。
5. 给生产赋码 `quantity` 和示例数据 `count` 增加上限。
6. 抽出 `AuthService`，降低 `AuthController` 厚度。
7. 修复前端当前红灯测试，并复核 router/request/store 循环依赖。
8. 修正文档中密钥、注册、当前用户接口、快速启动和权限迁移说明。

### P2：可维护性、文档债和工程卫生

1. 收敛 CORS 到单一机制，并拆分 dev/prod 默认值。
2. 抽取角色层级策略，避免 UserService / RoleService 重复硬编码。
3. 校验 `UserUpdateRequest.status`。
4. 评估 localStorage Token 风险，配合 CSP 和短 Token。
5. 清理根目录运行产物、大日志、大 CSV、运行截图。
6. 修复前端 API 注释乱码。
7. 重写 `frontend/README.md`。
8. 补启动路径集成测试：真实配置绑定、初始化 SQL、权限迁移、匿名注册、扫码契约、Redis 异常。

## 9. 修复验收建议

建议每一批修复完成后至少执行以下检查：

1. 后端：`mvn test -q` 全绿。
2. 前端：`npm run test:run` 全绿。
3. 手工或自动验证：
   - 未登录注册走 `/api/auth/register`。
   - `/api/users` 未登录不可访问。
   - 登出后 Token 立即失效；Redis 异常时行为符合设计。
   - 修改密码后旧 Token 失效。
   - 地图节点名包含 HTML 时不会作为 HTML 执行。
   - 删除已被溯源引用的配件会被拒绝。
   - 非法 eventTime 返回 400，不写入当前时间。
   - `remark` 的去留与 API 文档一致。
4. 文档检查：README、api-doc、postman-guide 与代码接口一致。
