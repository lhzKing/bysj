# 浏览器端全量测试任务清单（2026-05-11）

> **目的**：用 MCP 浏览器工具（chrome-devtools 或 playwright）对工业零配件供应链溯源系统（前端 https://localhost:5173 + 后端 http://localhost:8080）进行端到端测试，验证 **11 个前端路由 × 6 个角色 × 全部业务工作流 × 哈希链/RSA 验签** 的功能正确性，每个关键节点保留浏览器截图作为答辩证据。
>
> **执行者**：AI Agent（后续会话）。
> **数据集**：当前数据库由 `scripts/seed_extended_data.py` 一键铺底（详见 `数据库批量演示数据生成记录_20260511.md`），实测规模：12 用户 / 30 配件 / 18 节点 / 16 节点绑定 / 30 批次 / 611 trace_code / 611 snapshot / 3621 lifecycle_log / 60 流转任务 / 736 任务扫码明细 / 319 聚合关系（301 CARTON + 18 PALLET）。**全部 trace_lifecycle_log 都已通过 `verify_chains.py` 端到端验签**（hash_ok=125/125，signature_ok=125/125）。**测试用例尽量使用现有样本**，写操作会在表里明确"会产生新数据"标记。

## 零、AI 测试者强制执行准则（**最高优先级，必读**）

用户启动测试时只会发一句话「按照浏览器测试任务清单执行」。看到这个指令后，AI 必须按下列规则**一次性、不间断**跑完全部 15 个模块，不允许在任何模块边界停下来等待人工确认。

### 0.1 自动连贯执行

1. **不要在任何模块结束后停下**询问"是否继续下一模块"。整份清单视作一个原子任务，从模块 01 一直跑到模块 15，再写测试报告，然后才结束。
2. **不要在每个用例后**写很长的状态汇报。每条用例只在内部 TodoWrite 里更新进度即可，不要把详细叙述写到对话流；对话流仅在重要里程碑（每模块完成时 1-2 句）打点。
3. **不需要批准就可以做的事**（不要再问）：登录任意测试账号、访问任意前端路由、对测试数据做写操作（新建配件/用户/批次/任务/扫码记录）、调整浏览器 viewport、查询数据库读数据。
4. **需要先暂停一次问用户**的事（只有这两种）：(a) 准备模块 13.4 篡改 DB 日志前的最终确认（有数据破坏风险）；(b) 后端/前端服务挂掉无法在 2 分钟内自动恢复。其他所有情况都**自行决策、继续推进**。

### 0.2 失败不阻塞

- 单个用例失败不要中止整轮。立刻截图（页面 + console + network）→ 记录为 FAIL → 立刻进入下一用例。
- 失败用例集中体现在最终的`测试执行报告.md`里，不要在对话流中一项项报告。
- 一个模块里 ≥50% 用例失败时，**仍然继续**下个模块（可能是上游模块导致连锁失败，不是测试者该解决的）。

### 0.3 截图归档纪律

- 每个用例**完成截图后立即写盘**到对应模块目录，不要等到模块结束再批量保存。
- 截图文件名严格按 `{模块号}-{用例号}-{步骤号}-{中文描述}.png`，命名错了不要等用户纠正，自行覆盖修正。
- 单次截图占内存大，**不要**把截图二进制返回到对话流；只在 MCP 调用里给 `filePath`/`savePath`。

### 0.4 进度跟踪

- **必须用 TodoWrite**：开测前一次性把 15 个模块作为 15 个 todo 项写入，每完成一个模块立即 mark completed，再 mark 下一个 in_progress。
- 不需要把 127 条用例都写进 TodoWrite（太碎），只在模块粒度跟踪。

### 0.5 时间预算

- 全部 127 条用例预计 3-4 小时跑完。
- 如果跑到 6 小时还没完成 → 停下，把剩余用例标 SKIP，写最终报告。
- 不要为了追求"完美 PASS"反复重试同一用例。一次失败就记录，不再重试（除非是页面没加载完导致的 timeout，这种允许 1 次重试）。

## 五、AI 测试者操作准则

> 这一节是对 **"模块 0 强制执行准则"** 的细节补充。冲突时以模块 0 为准。

1. **开测时**：先创建模块粒度的 TodoWrite（15 项），把模块 01 标 in_progress，再开始执行。
2. **每个用例开始前**必须先把浏览器导航到正确路径，等待 `mcp__chrome-devtools__wait_for` 或 `mcp__playwright__browser_wait_for` 关键元素出现。
3. **避免随机睡眠**：用 `wait_for` 等元素，不要硬等几秒。
4. **失败立刻截图 + console + network 三连**，记录到报告后**立即进入下一用例**，不要试图重试或修复。
5. **写操作的新建数据用时间戳后缀**避免重复执行冲突（如 `tester_20260511_153022`）。
6. **数据库直接验证**：每个写操作完成后，可选用 mysql cli 验证数据落库（不强制截图）。
7. **不修改任何源码**。如果发现 bug，写到测试报告的"待修复"列，不要试图修。
8. **测试结束后**：不删测试期间新建的用户/批次/任务，留作答辩演示证据。
9. **跑完所有 15 个模块后**才停下来跟用户对话，输出最终汇总。**不允许在中间停下来征询意见**。
