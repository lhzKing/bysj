# 工业企业配件供应链溯源系统数据库设计

## 1. 设计目标

数据库围绕“全链路可追溯、状态可快速查询、权限可精细控制、数据可验真防篡改、批次赋码与任务驱动流转可落地”五个目标设计：

- 以**事实表 + 快照表**组合兼顾写入审计与查询性能。
- 以 **RBAC 权限模型**支撑多角色协同操作。
- 以 **Hash 链 + RSA 签名字段**支撑日志级防篡改验证。
- 以 **乐观锁版本号**保障并发扫码时状态更新一致性。
- 以 **赋码批次 + 单品码状态 + 结构化节点 + 流转任务 + 聚合关系**支撑“一物一码、批量赋码、快递式发货/接收、箱码/托盘码批量操作”。

---

## 2. 总体ER关系（文字版）

核心实体及关系如下：

1. **sys_role（角色）** 与 **sys_user（用户）**：一对多
   - 一个角色可绑定多个用户。
2. **sys_role（角色）** 与 **sys_permission（权限）**：多对多
   - 通过中间表 **sys_role_permission** 建立关联。
3. **base_part_spec（配件规格/SPU）** 与 **trace_assign_batch（赋码批次）**：一对多
   - 一个配件规格可被多个生产工单/赋码批次引用。
4. **trace_assign_batch（赋码批次）** 与 **trace_code（单品码）**：一对多
   - 一个批次一次性生成多个唯一单品码；单品码是追溯身份，批次是管理容器。
5. **base_part_spec（配件规格/SPU）** 与 **trace_snapshot（溯源快照）**：一对多
   - 一个配件规格可对应多条溯源码实例。
6. **trace_node（业务节点）** 与 **trace_user_node_binding（用户-节点绑定）**：一对多
   - 一个节点可授权给多个用户；用户可绑定多个可操作节点。
7. **trace_node（业务节点）** 与 **trace_flow_task（流转任务）**：一对多
   - 任务固定起点、终点和预计数量，扫码时后端按任务推导节点。
8. **trace_flow_task（流转任务）** 与 **trace_flow_task_scan（任务扫码明细）**：一对多
   - 支撑连续扫码、重复码反馈和实扫数量累计。
9. **trace_aggregation（聚合关系）** 自关联式码层级：
   - `parent_code` 可为箱码/托盘码/批量聚合码，`child_code` 可为单品码或下级聚合码。
10. **trace_snapshot（溯源快照）** 与 **trace_lifecycle_log（生命周期日志）**：一对多
    - 快照记录“当前状态”，日志记录“全量历史事件”。
11. **trace_lifecycle_log（生命周期日志）** 自关联：
   - 字段 `correction_of` 外键指向同表 `id`，用于红冲蓝补/修正链路。
12. **trace_scan_idempotency（扫码幂等记录）** 与 **trace_lifecycle_log（生命周期日志）**：多对一
    - `(trace_code, action_type, idempotency_key)` 唯一，重复提交返回第一次成功结果，不重复落日志。

## 3. 表结构设计

> 以下字段来自 `backend/sql/init_schema.sql` 与当前后端实体实现。

### 3.1 sys_role（角色表）

**用途**：定义系统角色（超级管理员、系统管理员、生产人员、仓库人员、物流人员、普通用户）。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| role_code | VARCHAR(32) | UNIQUE, NOT NULL | 角色代码（如 SUPER_ADMIN） |
| role_name | VARCHAR(64) | NOT NULL | 角色名称 |
| remark | VARCHAR(255) |  | 备注 |
| create_time | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |

---

### 3.2 sys_permission（权限表）

**用途**：定义权限点，包括接口方法、路径模式和权限语义。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| perm_code | VARCHAR(64) | UNIQUE, NOT NULL | 权限代码（如 trace:create） |
| perm_name | VARCHAR(64) | NOT NULL | 权限名称 |
| api_method | VARCHAR(10) |  | HTTP 方法（GET/POST/.../*） |
| api_pattern | VARCHAR(128) |  | API 路径模式（支持 `*`） |
| remark | VARCHAR(255) |  | 备注 |
| create_time | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |

---

### 3.3 sys_role_permission（角色权限关联表）

**用途**：建立 RBAC 多对多关系。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| role_id | BIGINT | PK(联合), FK | 角色ID |
| permission_id | BIGINT | PK(联合), FK | 权限ID |

**约束**：
- `fk_rp_role`: `role_id -> sys_role.id`（ON DELETE CASCADE）
- `fk_rp_perm`: `permission_id -> sys_permission.id`（ON DELETE CASCADE）

---

### 3.4 sys_user（用户表）

**用途**：保存登录主体、角色绑定与安全状态。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| username | VARCHAR(64) | UNIQUE, NOT NULL | 用户名 |
| password | VARCHAR(128) | NOT NULL | BCrypt 密文密码 |
| role_id | BIGINT | NOT NULL, FK | 角色ID |
| token_version | INT | NOT NULL, DEFAULT 0 | Token版本号（改密/改角色递增） |
| status | TINYINT | NOT NULL, DEFAULT 1 | 1=启用，0=禁用 |
| create_time | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**设计要点**：
- `token_version` 与 JWT 联动，实现“全量旧 token 失效”。
- `status` 实现用户启停控制。

### 3.5 base_part_spec（配件规格表，SPU）

| remark | VARCHAR(255) |  | 备注 |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

---

### 3.6 trace_node（结构化业务节点表）

**用途**：管理工厂、仓库、物流、客户、服务网点等真实业务节点，替代手填节点名称作为权限判断和任务流转的基础资料。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 节点主键 |
| node_code | VARCHAR(64) | UNIQUE, NOT NULL | 节点编码 |
| node_name | VARCHAR(64) | NOT NULL | 节点名称 |
| node_type | VARCHAR(32) | NOT NULL, CHECK | `FACTORY/WAREHOUSE/LOGISTICS/CUSTOMER/SERVICE` |
| org_id | BIGINT | NULL | 预留组织ID |
| province | VARCHAR(32) | NOT NULL | 省份 |
| city | VARCHAR(32) | NOT NULL | 城市 |
| address | VARCHAR(255) | NULL | 详细地址 |
| enabled | TINYINT(1) | NOT NULL, DEFAULT 1, CHECK | 1=启用，0=停用 |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**索引与约束**：
- `uk_trace_node_code(node_code)`：节点编码唯一。
- `idx_trace_node_type(node_type)`、`idx_trace_node_enabled(enabled)`：节点类型/启停筛选。
- `idx_trace_node_region(province, city)`：区域筛选。

---

### 3.7 trace_user_node_binding（用户-节点绑定表）

**用途**：保存用户可代表操作的业务节点，支撑扫码动作和可执行动作的节点级授权。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 绑定主键 |
| user_id | BIGINT | NOT NULL, FK | 用户ID |
| node_id | BIGINT | NOT NULL, FK | 业务节点ID |
| org_id | BIGINT | NULL | 从 `trace_node.org_id` 冗余复制，预留组织治理 |
| default_node | TINYINT(1) | NOT NULL, DEFAULT 0, CHECK | 1=默认操作节点 |
| enabled | TINYINT(1) | NOT NULL, DEFAULT 1, CHECK | 1=绑定启用 |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**索引与约束**：
- `uk_trace_user_node_binding(user_id, node_id)`：同一用户不可重复绑定同一节点。
- `fk_trace_user_node_user`：`user_id -> sys_user.id`（ON DELETE CASCADE）。
- `fk_trace_user_node_node`：`node_id -> trace_node.id`（ON DELETE CASCADE）。

---

### 3.8 trace_assign_batch（赋码批次表）

**用途**：把“生产赋码”从一次生成散列码改为可对账的生产/工单批次。批次只作为管理单位，单品码仍在 `trace_code.trace_code` 中保持一物一码。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 赋码批次ID |
| batch_no | VARCHAR(64) | UNIQUE, NOT NULL | 赋码批次号 |
| production_order_no | VARCHAR(64) | NULL | 生产计划/工单号 |
| spu_id | BIGINT | NOT NULL, FK | 关联SPU |
| quantity_requested | INT | NOT NULL, CHECK > 0 | 计划/请求生成数量 |
| quantity_generated | INT | NOT NULL, DEFAULT 0, CHECK | 已生成数量 |
| quantity_printed | INT | NOT NULL, DEFAULT 0, CHECK | 已打印的唯一码数量 |
| quantity_activated | INT | NOT NULL, DEFAULT 0, CHECK | 已激活数量 |
| manufacturer_node_id | BIGINT | NULL, FK | 生产节点 |
| status | VARCHAR(32) | NOT NULL, DEFAULT CREATED | `CREATED/GENERATING/GENERATED/PARTIAL_FAILED/FAILED/CANCELLED` |
| operator_id | BIGINT | NULL, FK | 创建操作人ID |
| operator_username | VARCHAR(64) | NULL | 创建操作人用户名 |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**设计要点**：
- `quantity_generated/printed/activated` 都受 `<= quantity_requested` 约束保护。
- 生产赋码接口返回 `batch_id/batch_no/requested_count/generated_count/trace_codes/batch_status/partial_failure/warning`，便于前端对部分失败做清晰展示。

---

### 3.9 trace_code（单品码状态表）

**用途**：保存单品码从“生成、打印、激活、在库、运输、异常、作废”到终态的标签/实物状态。它与 `trace_snapshot.current_status` 的区别是：`trace_code.code_status` 管理“码/标签是否可用”，`trace_snapshot.current_status` 管理“货物当前流转位置与物流状态”。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| trace_code | VARCHAR(64) | PK | 单品溯源码 |
| batch_id | BIGINT | NULL, FK | 所属赋码批次；历史码允许为空 |
| spu_id | BIGINT | NOT NULL, FK | 关联SPU |
| serial_no | INT | NULL, CHECK | 批次内序号 |
| qr_payload | VARCHAR(512) | NOT NULL | 二维码载荷 |
| code_status | VARCHAR(32) | NOT NULL, DEFAULT GENERATED | `GENERATED/PRINTED/ACTIVATED/IN_STOCK/IN_TRANSIT/EXCEPTION/VOIDED/SCRAPPED` |
| print_count | INT | NOT NULL, DEFAULT 0, CHECK >= 0 | 打印/重打次数 |
| activated_time | DATETIME | NULL | 激活时间 |
| activated_by | BIGINT | NULL, FK | 激活操作人ID |
| activated_by_username | VARCHAR(64) | NULL | 激活操作人用户名 |
| current_snapshot_id | VARCHAR(64) | NULL | 当前快照指针，当前等于 `trace_snapshot.trace_code` |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**索引与约束**：
- `uk_trace_code_batch_serial(batch_id, serial_no)`：批次内序号唯一。
- `idx_trace_code_status(code_status)`：按码状态筛选打印/激活/作废清单。
- `fk_trace_code_batch`、`fk_trace_code_spu`、`fk_trace_code_activated_by`：批次、SPU、激活用户引用。

---

### 3.10 trace_flow_task（流转任务/发货单表）

**用途**：以类似“快递单/发货任务”的方式固定起点、终点、预计数量和任务状态，让仓库/物流人员在任务内连续扫码，而不是每次手填 `from_node/to_node`。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 任务ID |
| task_no | VARCHAR(64) | UNIQUE, NOT NULL | 任务编号 |
| task_type | VARCHAR(32) | NOT NULL, CHECK | `OUTBOUND/TRANSFER/INBOUND/RECEIVE` |
| source_node_id | BIGINT | NOT NULL, FK | 起点节点 |
| target_node_id | BIGINT | NOT NULL, FK | 目标节点 |
| expected_quantity | INT | NOT NULL, CHECK > 0 | 应扫/预计数量 |
| actual_quantity | INT | NOT NULL, DEFAULT 0, CHECK >= 0 | 实扫/已处理数量 |
| status | VARCHAR(32) | NOT NULL, DEFAULT CREATED, CHECK | `CREATED/PROCESSING/COMPLETED/CANCELLED/EXCEPTION` |
| create_by | BIGINT | NULL, FK | 创建人ID |
| create_by_username | VARCHAR(64) | NULL | 创建人用户名 |
| complete_time | DATETIME | NULL | 完成时间 |
| cancel_time | DATETIME | NULL | 取消时间 |
| discrepancy_type | VARCHAR(32) | NOT NULL, DEFAULT NONE, CHECK | `NONE/SHORTAGE/OVERAGE` |
| discrepancy_quantity | INT | NOT NULL, DEFAULT 0, CHECK >= 0 | 差异数量 |
| discrepancy_reason | VARCHAR(255) | NULL | 少扫/多扫原因，数量不一致时必填 |
| discrepancy_time | DATETIME | NULL | 差异记录时间 |
| remark | VARCHAR(255) | NULL | 任务备注 |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**设计要点**：
- `source_node_id <> target_node_id`，避免无意义自流转。
- 数量一致可直接完成；数量不一致必须填写 `discrepancy_reason` 并将任务标记为异常。

### 3.11 trace_flow_task_scan（任务扫码明细表）

**用途**：保存任务内每个单品码的成功扫码事实，支撑连续扫码、重复码提示、父码展开后的批量统计。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 扫码明细ID |
| task_id | BIGINT | NOT NULL, FK | 流转任务ID |
| trace_code | VARCHAR(64) | NOT NULL | 被扫描/展开的单品码 |
| action_type | VARCHAR(32) | NOT NULL, CHECK | `OUTBOUND/INBOUND/TRANSFER` |
| counted | TINYINT(1) | NOT NULL, DEFAULT 1, CHECK | 1=计入任务实扫数量 |
| operator_user_id | BIGINT | NULL, FK | 扫码人ID |
| operator_username | VARCHAR(64) | NULL | 扫码人用户名 |
| idempotency_key | VARCHAR(64) | NULL | 任务扫码幂等键 |
| scan_time | DATETIME | NOT NULL | 扫码业务时间 |
| duplicate_count | INT | NOT NULL, DEFAULT 0, CHECK >= 0 | 首次成功后的重复扫码次数 |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**索引与约束**：
- `uk_trace_flow_task_scan_code_action(task_id, trace_code, action_type)`：同一任务同一动作下单品码只成功计数一次。
- `idx_trace_flow_task_scan_task_time(task_id, scan_time)`：任务详情和扫码时间线查询。

### 3.12 trace_aggregation（箱码/托盘码聚合关系表）

**用途**：保存箱码、托盘码、批量聚合码与子码之间的启用/解除关系。聚合码只是批量操作入口，不替代单品溯源码。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 聚合关系ID |
| parent_code | VARCHAR(64) | NOT NULL | 父码：箱码/托盘码/批量聚合码 |
| child_code | VARCHAR(64) | NOT NULL | 子码：单品码或下级聚合码 |
| relation_type | VARCHAR(32) | NOT NULL, CHECK | `CARTON/PALLET/BATCH` |
| active | TINYINT(1) | NOT NULL, DEFAULT 1, CHECK | 1=当前有效，0=已解除 |
| active_marker | TINYINT | GENERATED | 仅用于唯一约束的生成列 |
| create_by | BIGINT | NULL, FK | 绑定操作人ID |
| create_by_username | VARCHAR(64) | NULL | 绑定操作人用户名 |
| bind_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 绑定时间 |
| release_time | DATETIME | NULL | 解除时间 |
| remark | VARCHAR(255) | NULL | 备注 |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**索引与约束**：
- `uk_trace_aggregation_active_pair(parent_code, child_code, active_marker)`：同一父子对只能存在一条有效关系；历史解除关系可保留多条。
- `idx_trace_aggregation_parent_active(parent_code, active)`：扫父码展开子码。
- `idx_trace_aggregation_child_active(child_code, active)`：单品详情展示直接/间接聚合历史。

### 3.13 trace_snapshot（溯源快照表）

**用途**：保存每个溯源码实例的当前状态，支撑高频查询。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| trace_code | VARCHAR(64) | PK | 溯源码 |
| spu_id | BIGINT | NOT NULL | 关联SPU |
| current_status | VARCHAR(32) | NOT NULL | 当前状态 |
| current_node | VARCHAR(64) |  | 当前节点 |
| current_owner | VARCHAR(64) |  | 当前责任方 |
| exception_restore_status | VARCHAR(32) | NULL | 异常冻结前状态，`EXCEPTION_CLOSE` 用于恢复 |
| exception_restore_node | VARCHAR(64) | NULL | 异常冻结前节点 |
| exception_restore_owner | VARCHAR(64) | NULL | 异常冻结前责任方 |
| province | VARCHAR(32) |  | 省份（地图统计） |
| city | VARCHAR(32) |  | 城市 |
| last_event_time | DATETIME | NOT NULL | 最新业务时间 |
| last_log_id | BIGINT | NOT NULL | 最新日志ID |
| last_hash | CHAR(64) | NOT NULL | 链尾哈希（下一条 prev_hash） |
| version | INT | NOT NULL, DEFAULT 0 | 乐观锁版本号 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**设计要点**：
- 使用 `version`（MyBatis-Plus `@Version`）防并发覆盖。
- `last_hash` 将快照与日志链连接，保障链式连续性。
- `exception_restore_*` 只在异常冻结期间保存冻结前快照；解除异常后恢复原状态并清空恢复字段。

### 3.14 trace_lifecycle_log（溯源生命周期日志表）

**用途**：记录每次流转事件的全量事实，是审计、验链、追责的核心表。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 日志主键 |
| trace_code | VARCHAR(64) | NOT NULL | 溯源码 |
| spu_id | BIGINT | NOT NULL | 关联SPU |
| action_type | VARCHAR(32) | NOT NULL | 事件类型（见下方动作枚举） |
| from_node | VARCHAR(64) |  | 上游节点 |
| to_node | VARCHAR(64) |  | 下游节点 |
| province | VARCHAR(32) |  | 省份 |
| city | VARCHAR(32) |  | 城市 |
| event_time | DATETIME | NOT NULL | 业务时间（Event Time） |
| ingest_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 入库时间（Ingest Time） |
| prev_hash | CHAR(64) | NOT NULL | 前一条日志哈希 |
| current_hash | CHAR(64) | NOT NULL | 当前日志哈希 |
| correction_of | BIGINT | FK(NULL) | 被修正的原日志ID |
| operator | VARCHAR(64) | NOT NULL | 操作人；新写入日志已纳入 Hash/签名保护 |

## 4. 安全与一致性设计说明

### 4.1 防篡改机制

每条日志保存：
- `prev_hash`：上一条日志哈希
- `current_hash`：当前日志哈希
- `signature`：对标准化拼接数据的 RSA 签名
- `signature_key_id` / `signature_key_version`：签名密钥元数据，验签时优先匹配日志声明的密钥

系统可对链进行三类校验：
1. 链连续性校验（`prev_hash` 是否连续）
2. 哈希重算校验（`current_hash` 是否一致）
3. 签名验证（`signature` 是否可通过公钥验证）

新写入日志的 Hash/签名载荷包含 `operator` 和非空 `remark`；验链服务兼容历史未保护 `operator` 的旧载荷。

### 4.2 并发控制机制

- `trace_snapshot.version` 启用乐观锁。
- 流转更新使用独立事务 + 失败重试策略。
- 冲突超限后返回并发冲突错误，避免脏写。
- 普通扫码接口使用 `trace_scan_idempotency` 防重复提交。
- 任务内连续扫码使用 `trace_flow_task_scan` 唯一约束防重复计数，并记录 `duplicate_count` 作为重复扫码反馈。

### 4.3 权限控制机制

- `sys_user` + `sys_role` + `sys_permission` + `sys_role_permission` 构成标准 RBAC。
- 支持权限继承（如 `trace:inbound -> trace:scan -> trace:view`）。
- 支持细粒度动作权限（入库/出库/流转分离）。
- 支持新版业务动作权限：`trace:batch:create`、`trace:code:print`、`trace:code:activate`、`trace:task:create`、`trace:task:scan`、`trace:task:complete`、`trace:audit:view`、`trace:exception:handle`。
- 支持节点授权：用户必须绑定对应 `trace_node` 才能执行节点相关扫码动作；超级扫码/管理权限按策略放行。

## 5. 典型业务数据流与表写入

### 5.1 生产赋码

1. 写入 `trace_assign_batch`，记录批次号、工单号、SPU、计划数量、生产节点和创建人。
2. 为批次生成 N 个唯一 `trace_code`，每个码写入：
   - `trace_code`（`code_status=GENERATED`，保存批次、序号和二维码载荷）
   - `trace_lifecycle_log`（`INIT`，`prev_hash=GENESIS`，生成 Hash + RSA 签名）
   - `trace_snapshot`（当前状态兼容为 `INIT`，保存链尾日志与 Hash）
3. 更新 `trace_assign_batch.quantity_generated/status`；若分片失败，仅返回已成功落库的码并将批次标记为 `PARTIAL_FAILED`。
4. 标签打印、重打、作废和激活分别写入 `PRINT_CODE/REPRINT_CODE/VOID_CODE/ACTIVATE_CODE` 生命周期日志，并同步 `trace_code.print_count/code_status/activated_*` 与批次数量字段。

### 5.2 扫码流转

1. 若请求提供 `idempotency_key`，先占用 `trace_scan_idempotency(trace_code, action_type, idempotency_key)`。
2. 读取 `trace_snapshot` 当前版本和 `trace_code.code_status`。
3. `TraceTransitionPolicy` 校验“当前状态 + 动作 + 纠错关系”是否合法；未激活、作废、异常冻结等状态禁止常规流转。
4. 后端推导来源节点：常规场景 `from_node` 默认来自当前快照或任务起点；若前端伪造不一致来源节点则拒绝。
5. 插入 `trace_lifecycle_log` 新事件。
6. 更新 `trace_snapshot` 当前状态/节点/链尾哈希（带 version 条件），同步 `trace_code` 的物流态镜像。
7. 首次成功后回填幂等记录的 `lifecycle_log_id/status`；重复请求不再新建生命周期日志。

### 5.3 任务驱动流转

1. 管理员或业务人员创建 `trace_flow_task`，固定 `task_type/source_node_id/target_node_id/expected_quantity`。
2. 仓库/物流用户在任务内调用 `POST /api/trace-flow-tasks/{id}/scan` 连续扫码：
   - 扫单品码：写入一条任务扫码明细并追加对应生命周期事件。
   - 扫箱码/托盘码：从 `trace_aggregation` 展开有效子码，逐个执行同一任务动作；任一异常子码会整批拒绝。
   - 重复码：不重复计数，返回 `duplicate_scan/batch_duplicate_quantity` 等反馈字段。
3. 完成任务时，若 `actual_quantity == expected_quantity` 则状态为 `COMPLETED`；少扫/多扫必须填写差异原因并写入 `discrepancy_*` 字段，任务状态进入 `EXCEPTION`。

### 5.4 聚合码与单品详情

1. 装箱/托盘绑定写入 `trace_aggregation(active=1)`，并对受影响单品写入 `PACK/PALLETIZE` 生命周期日志。
2. 拆箱/托盘解绑将关系置为 `active=0`、写入 `release_time`，并写入 `UNPACK/UNPALLETIZE` 日志。
3. 已出库、运输中、已交接、异常冻结的单品禁止随意修改聚合关系。
4. `GET /api/traces/{traceCode}` 返回 `aggregation_history`，普通详情也能看到单品曾经所在箱/托盘；`view=audit` 同时展示完整聚合事件日志。

### 5.5 验链查询

1. 按 `trace_code` 拉取完整日志链
2. 逐条执行链连续、hash重算、签名验证
3. 返回验证结果与错误明细

## 6. 规范与优化建议

1. **历史兼容说明**：`trace_snapshot.current_node/current_owner` 与 `trace_lifecycle_log.from_node/to_node` 仍保留文本字段，用于历史数据和展示；新业务判断优先使用 `trace_node`、`trace_user_node_binding` 与 `trace_flow_task`。
2. **归档策略**：日志表数据量增长快，可按月分区或冷热分层。
3. **索引维护**：统计查询与写入冲突时，定期评估二级索引数量。
4. **密钥治理**：生产环境固定管理 RSA 私钥，禁用自动生成。
5. **审计扩展**：当前已有 `operator`、`signature_key_id/version`、`idempotency_key`、任务扫码操作人等字段；后续可继续新增操作终端、IP、请求ID等字段，提升取证能力。

## 7. 小结

本数据库方案采用“权限域 + 配件域 + 批次/单品码域 + 节点/任务域 + 聚合码域 + 溯源事实域”分层建模，既支持工业供应链场景下的高并发流转记录，又能通过哈希链与数字签名确保数据可验真、可追责。整体结构与当前系统实现一致，能够覆盖批次赋码、一物一码、扫码激活、任务驱动出入库、箱码/托盘码批量流转、异常冻结和红冲蓝补审计等新版核心业务。
