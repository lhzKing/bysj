## 摘  要

随着制造业数字化转型不断推进，工业企业对配件供应链的透明化、可追踪与可追责能力提出了更高要求。针对传统管理模式中“来源难核验、流转难追踪、异常难定位、责任难界定”等问题，本文设计并实现了一套基于 Spring Boot 的工业企业配件供应链溯源系统。

系统采用前后端分离架构：后端基于 Spring Boot、MyBatis-Plus、MySQL、Redis 构建统一业务服务与安全基础设施；前端采用 Vue 3、Vite、ECharts 实现多端可视化交互。围绕“配件基础档案—生产赋码—扫码流转—链路验真—统计分析”构建完整业务闭环。在数据建模上，采用“快照表 + 生命周期日志表”的双表策略：快照表支持高频状态查询，日志表保留完整事实链用于审计追溯。在安全机制上，系统融合 JWT 认证、RBAC 授权、Token 黑名单和 Token 版本控制，支持细粒度权限继承与接口级权限匹配。在数据可信方面，引入 Hash 链与 RSA 数字签名机制，实现日志级防篡改与第三方可验真。针对扫码高并发写入场景，系统基于乐观锁与重试策略保障状态一致性。

项目实践结果表明，该系统能够有效支撑工业配件从生产到仓储、物流、使用等多节点全生命周期追踪，具备较好的可维护性、扩展性与工程落地价值，可为中小制造企业供应链数字化治理提供可复用方案。

**关键词**：工业供应链；配件溯源；Spring Boot；RBAC；数字签名；Hash链

## Abstract

With the continuous digital transformation of manufacturing, industrial enterprises require stronger capabilities in transparency, traceability, and accountability across spare-part supply chains. To address common problems in traditional management, such as unverifiable origins, opaque circulation, slow anomaly localization, and unclear responsibility, this thesis designs and implements an industrial spare-part supply chain traceability system based on Spring Boot.

The system adopts a front-end/back-end separated architecture. The backend is built with Spring Boot, MyBatis-Plus, MySQL, and Redis to provide unified business services and security infrastructure. The frontend uses Vue 3, Vite, and ECharts for multi-terminal visualization and interaction. A complete business loop is established around part master data, code assignment, scan-based circulation, chain verification, and dashboard analytics. For data modeling, a dual-table strategy is used: a snapshot table for high-frequency status queries and a lifecycle log table for full factual audit trails. In terms of security, JWT authentication, RBAC authorization, token blacklist, and token version control are integrated, together with fine-grained permission inheritance and API-level permission matching. For data trustworthiness, hash chaining and RSA digital signatures are introduced to achieve anti-tampering and third-party verifiability. To handle high-concurrency scan writes, optimistic locking and retry mechanisms are applied to ensure state consistency.

Implementation results show that the system effectively supports full-lifecycle tracking of industrial parts across production, warehousing, logistics, and usage scenarios. It demonstrates good maintainability, extensibility, and practical deployment value, and can serve as a reusable solution for digital supply chain governance in small and medium-sized manufacturing enterprises.

**Keywords**: industrial supply chain; part traceability; Spring Boot; RBAC; digital signature; hash chain

## 第1章 绪论

### 1.1 研究背景

在装备制造、汽车零部件、机械加工等行业中，工业配件通常具有种类多、批次多、流转节点多、责任主体多等特点。传统以纸质单据或分散系统为主的管理方式普遍存在信息孤岛，难以支持跨节点实时追踪。一旦发生质量问题、错发漏发或库存异常，企业往往需要大量人工核对记录，排查周期长、成本高。

与此同时，企业数字化转型进入深水区，管理目标从“信息化记录”逐步升级为“过程可信、链路可证、责任可追”。这要求溯源系统不仅能够记录数据，还应提供防篡改能力、精细权限治理和可视化决策支持。因此，研究并实现一套可落地、可扩展的工业配件供应链溯源系统具有明确工程价值。

### 1.2 研究目的与意义

本课题以“低门槛部署、高可信追溯、可持续演进”为目标，围绕工业企业常见业务流程构建溯源平台。其目的包括：

1. 建立一物一码机制，实现配件从源头到末端的全流程可追踪；
2. 构建多角色协同下的细粒度权限控制，降低越权风险；
3. 构建可验证的数据完整性机制，提升溯源结果公信力；
4. 通过可视化看板增强管理层对供应链状态的感知与决策效率。

研究意义体现在：一是形成工业配件场景下的完整系统设计方法与实现样例；二是验证轻量级技术栈在真实业务复杂度下的可行性；三是为后续接入物联网设备、区块链存证或智能分析能力提供基础架构。

### 1.3 国内外研究现状

相关研究主要集中在三个方向：

1. **溯源架构与业务建模**：聚焦实体标识、事件流转和历史回放能力；
2. **可信机制**：采用哈希链、数字签名、分布式账本等手段保证数据真实性；
3. **行业化落地**：在食品、医药领域应用较多，在工业配件场景仍存在“模块化不足、系统割裂、可操作性偏弱”的问题。

从工程实践看，纯理论模型往往忽略企业现有技术栈兼容问题；重型可信架构虽安全性高，但在中小企业场景可能面临成本与实施复杂度偏高问题。本课题选择在现有 Java Web 工程体系中落地“可信溯源 + 权限治理 + 可视分析”，强调可部署性与演进性。

## 第2章 系统需求分析

### 2.1 业务场景分析

系统面向工业企业典型流转路径：生产赋码、入库、出库、跨节点转运、异常处理与修正。核心对象是“配件实例（trace_code）”，核心目标是保证每一次状态变更都可记录、可验证、可追责。

主要参与角色如下：

- 超级管理员（SUPER_ADMIN）：系统全局管理；
- 系统管理员（ADMIN）：用户角色管理与配置维护；
- 生产人员（PRODUCER）：执行生产赋码；
- 仓库人员（WAREHOUSE）：执行入库/出库操作；
- 物流人员（LOGISTICS）：执行流转操作；
- 普通用户（USER）：查询溯源与看板信息。

### 2.2 功能需求分析

#### 2.2.1 用户与权限管理

- 用户注册、登录、登出、密码修改；
- 用户启停、角色变更、重置密码；
- 角色权限分配与权限清单维护；
- 按角色动态授权接口访问。

#### 2.2.2 配件管理

- 配件基础档案管理（编码、名称、类别、型号、制造商）；
- 支持分页检索、按类型/厂商筛选；
- 提供赋码流程的基础数据来源。

#### 2.2.3 溯源管理

- 生产赋码：生成 trace_code 并建立初始链；
- 扫码流转：记录入库、出库、转运、异常、修正等事件；
- 溯源查询：查看当前快照与历史事件时间线；
- 验链查询：返回链路完整性与签名验证结果。

#### 2.2.4 可视化看板

- KPI：总追溯数、当日新增、日志总量、异常量；
- 地图：按省份聚合流转热度；
- 趋势：按时间窗口统计事件数量；
- 拓扑：节点间流向关系图。

### 2.3 非功能需求分析

1. **安全性**：必须具备身份认证、权限控制与数据防篡改能力；
2. **一致性**：并发扫描同一溯源码时不得出现状态覆盖；
3. **可维护性**：模块职责清晰，便于后续迭代；
4. **可扩展性**：支持后续新增节点类型、权限点和统计维度；
5. **易用性**：支持桌面端与移动端访问，扫码操作便捷。

### 2.4 可行性分析

- **技术可行性**：Spring Boot + Vue 技术栈成熟，组件生态完善；
- **经济可行性**：主要依赖开源框架，部署成本可控；
- **实施可行性**：可在毕业设计周期内完成核心功能与测试验证。

### 2.5 用例分析与验收标准

结合任务书与业务访谈，系统核心用例可概括为：

1. **生产赋码用例**：生产人员选择配件规格并生成溯源码，系统写入初始日志与快照；
2. **扫码流转用例**：仓库/物流人员执行入库、出库、转运动作，系统更新状态并写入事件日志；
3. **链路追溯用例**：管理员或普通用户按溯源码查询全生命周期事件；
4. **验链核验用例**：质量管理人员对日志链执行完整性与签名校验；
5. **运营分析用例**：管理层通过看板观察区域分布、趋势变化与异常占比。

为保证需求可验证，系统制定如下验收标准：

- 功能验收：上述核心用例均可在演示环境完成闭环；
- 安全验收：未授权访问必须被拦截，登录态可强制失效；
- 一致性验收：并发扫码不出现状态覆盖与链路错乱；
- 可视化验收：看板指标与日志统计结果一致，且支持按时间筛选。

## 第3章 系统总体设计

### 3.1 系统架构设计

系统采用前后端分离架构：

- 前端：Vue 3 + Vite + Pinia + Vue Router + ECharts；
- 后端：Spring Boot + MyBatis-Plus + MySQL + Redis；
- 通信方式：RESTful API（JSON）；
- 认证方式：JWT（Bearer Token）。

请求链路总体为：

客户端请求 → CORS过滤 → 登录拦截（Token校验）→ 权限拦截（RBAC校验）→ Controller → Service → Mapper → MySQL/Redis

### 3.2 功能模块设计

系统按职责划分为六个核心模块：

1. 认证模块（Auth）
2. 用户模块（User）
3. 角色权限模块（Role/Permission）
4. 配件模块（Part）
5. 溯源模块（Trace）
6. 看板模块（Dashboard）

各模块通过统一响应模型进行交互，降低前后端耦合。

### 3.3 数据库设计

数据库设计围绕“可追溯、可审计、可扩展、高并发一致性”四个目标展开，整体采用关系型模型并按业务域拆分为权限域、配件域、溯源域三层。

#### 3.3.1 E-R关系设计

核心实体关系如下：

1. `sys_role`（角色）与 `sys_user`（用户）为一对多关系；
2. `sys_role`（角色）与 `sys_permission`（权限）为多对多关系，通过 `sys_role_permission` 关联；
3. `base_part_spec`（配件规格）与 `trace_snapshot`（溯源快照）为一对多关系；
4. `trace_snapshot`（快照）与 `trace_lifecycle_log`（生命周期日志）为一对多关系；
5. `trace_lifecycle_log` 通过 `correction_of` 字段形成自关联，用于红冲蓝补和异常修正追踪。

该关系设计保证“权限控制、主数据维护、业务事实追踪”三类职责边界清晰，便于后续演进。

#### 3.3.2 核心数据表结构设计

（1）权限域表

- `sys_role`：角色主数据，记录 `role_code`、`role_name` 等；
- `sys_permission`：权限点定义，记录 `perm_code`、`api_method`、`api_pattern`；
- `sys_role_permission`：角色与权限映射关系，联合主键避免重复授权；
- `sys_user`：用户主体，包含 `token_version` 与 `status` 字段，支持会话强制失效与启停控制。

（2）配件域表

- `base_part_spec`：配件规格主表，记录编码、类别、型号、厂商、计量单位等，为赋码提供标准化基础信息。

（3）溯源域表

- `trace_snapshot`：保存每个 `trace_code` 的当前状态、当前节点、链尾哈希、最后日志ID与版本号（`version`）；
- `trace_lifecycle_log`：记录每次流转事件事实，包含 `action_type`、`from_node`、`to_node`、`event_time`、`prev_hash`、`current_hash`、`signature` 等关键字段。

“快照表 + 日志表”双表模式的设计动机为：快照满足高频查询性能，日志满足全链路审计与验真能力。

#### 3.3.3 关键约束与索引设计

1. 主键与唯一约束  
   - `sys_user.username`、`sys_role.role_code`、`sys_permission.perm_code`、`base_part_spec.part_code` 采用唯一约束；
   - `trace_snapshot.trace_code` 作为主键，确保“一码一快照”。

2. 外键与完整性约束  
   - `sys_user.role_id -> sys_role.id`；
   - `sys_role_permission.role_id -> sys_role.id`；
   - `sys_role_permission.permission_id -> sys_permission.id`；
   - `trace_lifecycle_log.correction_of -> trace_lifecycle_log.id`（可空外键）。

3. 索引设计  
   - `trace_lifecycle_log(trace_code)`：加速按码追溯；
   - `trace_lifecycle_log(event_time)`：支持时间窗口统计；
   - `trace_lifecycle_log(province)`：支持地图聚合；
   - `trace_lifecycle_log(from_node,to_node)`：支持拓扑分析；
   - `trace_snapshot(version)` 结合乐观锁策略保障并发写一致性。

#### 3.3.4 核心数据模拟与采集方案

为支撑中期检查与系统测试，项目完成了如下数据准备工作：

1. 基础主数据：模拟配件规格、分类、厂商、用户与角色权限集合；
2. 业务过程数据：按“生产-入库-出库-转运-异常-修正”生成完整生命周期事件；
3. 异常对照数据：人工构造断链、签名失效、跨链修正等样本用于验链测试；
4. 可视化数据：按省份、时间窗口与节点流向生成统计样本，驱动看板展示。

通过上述数据准备，系统可稳定演示核心业务流程并支撑功能测试、并发测试与验链测试。

### 3.4 接口规范设计

系统接口统一前缀为 `/api`，响应体采用统一结构：

```json
{
  "code": 0,
  "status": 200,
  "message": "success",
  "data": {}
}
```

主要规范：

- 请求字段兼容 `camelCase` 与 `snake_case`；
- 响应字段统一为 `snake_case`；
- 列表统一返回 `items + total`；
- 错误统一由全局异常处理器输出。

### 3.5 安全架构设计

#### 3.5.1 身份认证

- 用户登录后签发 JWT；
- Token 含 `role`、`token_version`、`jti` 等关键声明；
- 请求通过拦截器进行有效性校验。

#### 3.5.2 授权控制

- 使用 `@RequirePermission` 注解声明接口权限；
- 支持权限继承（如管理权限隐含查看权限）；
- 支持细粒度扫码权限（`trace:inbound`、`trace:outbound`、`trace:transfer`）。

#### 3.5.3 会话失效

- 登出后将 Token 的 `jti` 写入 Redis 黑名单；
- 用户改密/改角色时递增 `token_version`，强制旧 Token 失效。

#### 3.5.4 数据可信

- 每条日志写入 `prev_hash/current_hash/signature`；
- 系统提供公钥接口，支持第三方独立验签。

## 第4章 系统详细设计与关键实现

### 4.1 认证与权限模块实现

`AuthController` 实现登录、注册、登出、刷新 Token、获取当前用户信息、修改密码等接口。密码采用 BCrypt 存储。登录成功后返回 Token 与权限列表，供前端动态渲染按钮与路由。

`PermissionService` 作为权限门面，内部拆分为：权限查询、权限继承解析、API 匹配、缓存管理等组件，实现职责清晰的权限判断流程。

在权限继承中，系统定义了典型规则：

- `user:manage -> user:view`
- `role:manage -> role:view`
- `part:manage -> part:view`
- `trace:inbound/outbound/transfer -> trace:scan -> trace:view`

这使得权限体系既可精细控制，也能减少重复授权配置。

在认证鉴权实现中，系统采用“登录签发 + 请求拦截 + 注解校验”的三段式链路：

1. 登录阶段：校验账号状态、比对 BCrypt 密码、签发 JWT 并写入登录审计；
2. 请求阶段：通过拦截器解析 Token、校验过期时间、检查黑名单、校验 token_version；
3. 授权阶段：读取接口所需权限，结合角色权限及继承关系进行匹配，决定放行或拒绝。

该链路能够在保证安全性的同时，将鉴权逻辑与业务逻辑解耦，提升模块可维护性。

### 4.2 配件管理模块实现

配件管理围绕 `base_part_spec` 实体构建，支持：

- 分页查询（关键词、编码、类别、厂商）；
- 详情查询；
- 新增、修改、删除、批量删除；
- 类型与厂商下拉数据查询。

该模块为赋码流程提供统一 SPU 基础数据来源。

### 4.3 溯源赋码与流转模块实现

#### 4.3.1 赋码流程

在 `TraceCodeAssignmentService` 中执行：

1. 校验配件标识（`spuId`/`partCode`）；
2. 生成 `trace_code`；
3. 创建 INIT 日志（含 hash 与签名）；
4. 创建快照记录（初始化状态、链尾 hash）。

#### 4.3.2 扫码流转流程

在 `TraceScanTransactionService` 中执行事务性更新：

1. 查询快照；
2. 校验修正逻辑（防跨链修正攻击）；
3. 生成并插入新日志；
4. 按 action_type 推导新状态并更新快照；
5. 更新失败触发乐观锁冲突。

外层由 `TraceScanRetryExecutor` 承担重试控制，最多重试 3 次，并采用渐进退避（50ms × retryCount）。

#### 4.3.3 状态机与动作映射规则

系统将流转动作抽象为有限状态机，核心约束如下：

1. `INIT -> INBOUND -> STORED -> OUTBOUND -> IN_TRANSIT -> RECEIVED` 为主链路；
2. `EXCEPTION` 可由任意业务状态触发，但必须记录触发原因与责任节点；
3. `CORRECTION` 仅允许修正同一 `trace_code` 的目标日志，禁止跨链修正；
4. 非法状态迁移直接拒绝并返回业务错误码，避免脏数据进入链路。

该规则有效保证了状态转换的可解释性和可审计性。

#### 4.3.4 异常处理与幂等控制

为防止重复扫码、网络抖动重放、并发提交等问题，系统采用如下策略：

1. 请求幂等键：对关键扫码请求按 `trace_code + action_type + event_time` 进行去重；
2. 事务回滚：日志写入或快照更新任一环节失败均整体回滚；
3. 冲突重试：出现乐观锁冲突时自动重试，超过阈值返回明确提示；
4. 审计留痕：所有异常分支均写入日志与错误明细，支持后续问题复盘。

### 4.4 验链模块实现

`TraceChainVerifyService` 对指定 trace_code 执行完整验链，包含：

1. **链连续性校验**：`prev_hash` 与上一条 `current_hash` 是否一致；
2. **Hash 重算校验**：根据字段重算 `current_hash`；
3. **签名验证**：使用公钥验证 `signature`。

若存在断链、Hash 不匹配或签名无效，系统返回详细错误列表，便于定位异常日志。

为提升诊断效率，验链返回结构中包含失败日志ID、失败类型、预期值与实际值，运维人员可直接定位到异常事件并进行二次核验。

### 4.5 看板模块实现

看板模块由 `DashboardService` 和 `DashboardMapper.xml` 组成，支持按 `today/7d/30d/180d/all` 查询范围统计。

- KPI：聚合溯源码数量、日志量、异常量；
- 地图：按省份统计事件数；
- 趋势：按小时/天输出序列；
- 拓扑：按 `from_node -> to_node` 聚合边关系。

该模块可为企业管理者提供直观运营态势感知。

同时，看板支持筛选时间粒度（小时/天）与业务动作类型（入库、出库、转运、异常），便于不同管理角色聚焦关键指标。

### 4.6 前端交互与扫码实现

前端采用 Vue 3 组件化设计，按业务域划分 features（trace、part、user、dashboard）。扫码界面基于 `vue-qrcode-reader` 封装，支持：

- 摄像头权限检测与错误提示；
- 实时识别二维码内容；
- 入库/出库/流转动作模式切换；
- 移动端友好交互。

在 UI 层，页面结合表格、时间线、地图、拓扑等组件展示溯源全貌，提升可读性与操作效率。

### 4.7 关键实现难点与优化策略

在项目实现过程中，主要难点及解决策略如下：

1. **权限链复杂度高**：通过权限门面拆分与缓存策略降低权限计算开销；
2. **并发写冲突频发**：通过乐观锁 + 渐进退避重试保障高并发一致性；
3. **验链计算成本高**：通过链尾哈希与按需增量校验降低全链重算频次；
4. **前端跨端适配难**：采用统一组件规范与响应式布局提升移动端可用性。

上述优化使系统在可运行原型阶段即可达到较好的稳定性与可维护性，为后续工程化部署打下基础。

## 第5章 系统测试与结果分析

### 5.1 测试环境

- 操作系统：Windows
- 后端：JDK 17+/19、Spring Boot 3.2.x
- 前端：Node.js 18+、Vite 5+
- 数据库：MySQL 8+
- 缓存：Redis 7+

### 5.2 功能测试

围绕“用户管理、角色权限、配件管理、赋码、扫码、查询、验链、看板”进行接口与页面联调。结果表明，系统主流程均可正常闭环，响应数据与业务预期一致。

典型功能测试点包括：

1. 用户从登录到扫码流转的完整业务闭环；
2. 不同角色下菜单、按钮和接口权限的一致性；
3. 配件建档后赋码、扫码、追溯查询的数据一致性；
4. 看板统计数据与原始日志数据的对账一致性。

### 5.3 安全测试

1. 未登录访问受保护接口，返回 401；
2. 权限不足访问管理接口，返回 403；
3. 登出后旧 Token 无法继续访问；
4. 修改密码后历史 Token 失效。

测试说明认证授权链路有效，能够防止常见越权访问。

此外，通过伪造 Token、篡改 Token 载荷、重复使用历史 Token 的方式进行对抗性测试，系统均能拒绝非法请求并返回规范错误信息。

### 5.4 并发一致性测试

针对同一 trace_code 发起并发扫码请求，验证：

- 快照版本冲突可被检测；
- 冲突后可自动重试；
- 重试失败时返回并发冲突提示。

结果表明，系统可有效避免并发覆盖写问题，保证状态更新一致性。

在 50~100 并发用户的压测场景下，系统能够保持稳定响应，冲突请求在重试机制下可恢复成功，少量超限请求被正确识别并提示“请稍后重试”。

### 5.5 验链准确性测试

通过构造正常链与异常链（模拟 hash 或 signature 变更）进行对照测试：

- 正常链：返回 valid；
- 篡改链：准确输出断链/签名错误信息。

说明系统在数据真实性校验方面具备可用性。

同时，系统可输出逐条校验明细（日志ID、校验项、错误原因），能够满足审计场景对“可解释失败结果”的要求。

### 5.6 结果分析

系统在可用性、功能完整性和安全性方面达到预期目标。相比传统表格化记录方式，系统显著提升了问题定位效率与责任追溯能力。其不足在于当前缺少更精细的性能压测报告与跨企业协同标准接口，后续可继续完善。

综合测试结果可得：

1. 功能层面：核心业务链路完整，主要功能可稳定演示；
2. 技术层面：认证鉴权、并发控制、验链机制协同工作正常；
3. 管理层面：看板指标对运营分析具有辅助决策价值；
4. 改进方向：需进一步完善长期压测、容灾演练与多组织协同接口规范。
