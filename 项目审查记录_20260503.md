## 2. 做得好的地方

### 2.1 启动期配置守卫（强烈正向评价）
- **位置**：[backend/src/main/java/com/example/trace/config/ProdProfileConfigGuard.java:34-108](backend/src/main/java/com/example/trace/config/ProdProfileConfigGuard.java#L34)
- **优点**：用 `ApplicationContextInitializer` 在 Spring 上下文创建之前就拒绝错误配置——默认 JWT secret、`root/root` 数据库、`auto-generate=true`、占位符路径、指向 `backend/keys` 的工作区路径都会让 prod 启动失败。这种 *fail-fast* 模式比"运行时再发现配置错"靠谱得多。
- **正面影响**：把"开发污染生产"这一类历史漏洞做成了硬约束。

### 2.2 乐观锁的事务边界拆分
- **位置**：[TraceScanRetryExecutor.java:22-43](backend/src/main/java/com/example/trace/service/impl/support/TraceScanRetryExecutor.java#L22) 外层无 `@Transactional`，[TraceScanTransactionService.java:40](backend/src/main/java/com/example/trace/service/impl/support/TraceScanTransactionService.java#L40) 内层 `REQUIRES_NEW`
- **优点**：完全符合 [CLAUDE.md](CLAUDE.md) 描述的"重试不能放在同一事务"——旧事务必须真正回滚后，新的 `version` 才会对其它事务可见。`50ms*retry` 退避 + 最多 3 次 + 抛 `BizCode.CONCURRENT_CONFLICT`，整套链条干净。
- **正面影响**：避免"同事务里 while-retry 永远读到 stale version 死循环"这种典型新手坑。

### 2.3 TokenStore fail-closed
- **位置**：[TokenStore.java:96-99](backend/src/main/java/com/example/trace/security/TokenStore.java#L96) 抛 `TokenStoreException`，[LoginInterceptor.java:75-78](backend/src/main/java/com/example/trace/security/LoginInterceptor.java#L75) 返回 503
- **优点**：原本 Redis 故障会让所有"已登出 Token"瞬间复活，现在直接 503 让客户端重试。
- **正面影响**：把基础设施异常显式暴露，比静默放行安全得多。

### 2.4 Hash 链 + 签名密钥元数据

- **正面影响**：演示价值高（毕设答辩可解释），同时不阻塞未来扩展。

  4. 启用高德的"白名单域名"限制，缩小被滥用的爆炸半径。
- **是否建议立即处理**：是

## 4. 主要设计缺陷

### [主要] R-02 单事务内做大量 RSA 签名，长事务持锁
- **位置**：[TraceCodeAssignmentService.java:50-107](backend/src/main/java/com/example/trace/service/impl/support/TraceCodeAssignmentService.java#L50)（最大 quantity=500）；[TraceDemoDataServiceImpl.java:78-107](backend/src/main/java/com/example/trace/service/impl/TraceDemoDataServiceImpl.java#L78)（最大 count=500）；签名工具 [SignatureUtil.java:139-156](backend/src/main/java/com/example/trace/util/SignatureUtil.java#L139)
- **现象**：批量赋码/演示数据生成方法标了 `@Transactional`，循环里每条记录调一次 RSA 2048 签名 + 两次 INSERT。500 条意味着 500 次 SHA256withRSA 串行计算 + 1000 条 INSERT，全部在同一事务里。
- **影响**：
  - 单次请求耗时秒级（实测 RSA-2048 sign 在普通 CPU 约 1-3ms/次，500 条至少 0.5s 仅签名，加 IO 总耗时 2-5s 量级）；
  - 同时占用一条数据库连接 + 一个 HTTP 工作线程；
  - 长事务无法被取消，前端超时（[request.js:12](frontend/src/core/api/request.js#L12) 默认 10s）失败时事务可能仍在进行；
  - 高并发批量生产场景会快速耗尽 HikariCP 连接池。
- **原因分析**：把"签名"和"写库"放进同一原子单元是不必要的——签名本身是确定性计算，可以先批量算好，再用更小颗粒度提交。
- **修复建议**：
  1. 把签名移出 `@Transactional`，先在内存里组装好所有 `TraceLifecycleLog` 和 `TraceSnapshot`；
  2. 用 MyBatis-Plus `saveBatch` 或分页 commit（每 50/100 条一次事务）；
  3. 大于阈值（如 100）的请求改异步任务 + 任务进度查询接口；
  4. 演示数据生成接口（[AdminController.java:33-42](backend/src/main/java/com/example/trace/controller/AdminController.java#L33)）天然适合改成异步。
- **是否建议立即处理**：否，但应在下一个迭代里处理

### [主要] R-03 PermissionCache 是 JVM 本地缓存，无横向扩展能力
- **位置**：[security/permission/PermissionCache.java:17-50](backend/src/main/java/com/example/trace/security/permission/PermissionCache.java#L17)；写入侧 [PermissionService.java:112-119](backend/src/main/java/com/example/trace/security/PermissionService.java#L112)（`clearCache`）
- **现象**：用 `ConcurrentHashMap<Long, ...>` 做权限缓存，`clearCache` 仅清当前 JVM。
- **影响**：
  - 单机部署没问题；
  - 一旦多实例（比如 K8s 副本数 >1），管理员通过 A 实例修改了 ROLE 权限，B 实例的拦截器仍然按旧权限放行，可能持续到 B 重启；
  - 这是"权限变更后旧 token 被快速失效"安全模型的一个**隐性破口**。
- **原因分析**：缓存设计时未考虑分布式失效。架构注释里也没 disclaim 单实例假设。
- **修复建议**：
  1. 短期：在 README/CLAUDE.md 显式标注"目前权限模型假设单实例部署"；
  2. 中期：改用 Redis 缓存 + 在 [PermissionService.clearCache()](backend/src/main/java/com/example/trace/security/PermissionService.java#L112) 通过 Redis pub/sub 通知所有实例；
  3. 或者改成短 TTL（如 30s）的本地缓存，降低不一致窗口。
- **是否建议立即处理**：否（毕设阶段可接受），但需文档明示

### [主要] R-04 Token 存于 localStorage，XSS 即等于盗号
- **位置**：[frontend/src/core/auth/authStorage.js:1-15](frontend/src/core/auth/authStorage.js#L1)
- **现象**：JWT 写入 `localStorage.token`，[request.js:30-32](frontend/src/core/api/request.js#L30) 每次请求读取并放到 `Authorization` 头。
- **影响**：任何 XSS（比如未来引入第三方脚本被劫持、CDN 投毒）都能直接读取 token 后远程发起任何已登录用户的操作。
- **原因分析**：是前端 SPA 常见的"开发期方便"妥协。当前未观察到 XSS sink（已抽查 `v-html`、`innerHTML` 全无问题，InfoWindow 也是 DOM API），但纵深防御应该再加一层。
- **修复建议**：
  1. 服务端把 access token 改为 httpOnly + SameSite=Strict cookie；
  2. 前端不再持有原始 Token，请求自动带 cookie；
  3. 改造前**至少**给 README 加一段明确的威胁模型说明。
- **是否建议立即处理**：否（毕设范围下可接受），但属于已知风险

### [主要] R-05 Dashboard SQL 子查询会随数据增长退化
- **位置**：[backend/src/main/resources/mapper/DashboardMapper.xml:27-37](backend/src/main/resources/mapper/DashboardMapper.xml#L27) `selectKpi` 内嵌三个全表统计；[同文件:39-48](backend/src/main/resources/mapper/DashboardMapper.xml#L39) `selectMapData`、[97-111](backend/src/main/resources/mapper/DashboardMapper.xml#L97) `selectTopologyEdges` 都没有 LIMIT
- **现象**：
  - `selectKpi` 在主查询的 `trace_lifecycle_log` 上 `COUNT(*)` 同时套两个针对 `trace_snapshot` 的子查询（`COUNT(DISTINCT trace_code)`）；
  - `selectTopologyEdges` 没有 LIMIT，topology 边可能超过千条。
- **影响**：随 `trace_lifecycle_log` 行数增长，仪表盘加载从毫秒退化到秒级。
- **修复建议**：
  1. KPI 改为基于时间窗的查询并加 `idx_event_time`、`idx_create_time`；
  2. 拓扑边加 `LIMIT 200` 之类的上限并按权重排序；
  3. 长期可加 5 分钟级的 KPI 缓存或物化视图。
- **是否建议立即处理**：否，但若数据量真上升应优先处理
