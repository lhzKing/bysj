# T17 PrimeVue 登录页与主布局重构设计

**日期**：2026-04-11  
**任务编号**：T17  
**任务名称**：重构登录页与主布局为 PrimeVue 风格

## 1. 目标

在 T14-T16 已完成 PrimeVue 基线、主题令牌和基础 UI 组件层收口的前提下，T17 聚焦系统入口页与全局框架层，完成以下目标：

- 将登录页从“普通表单页”升级为具备毕业设计展示完成度的企业级 SaaS 单卡片登录页
- 将主布局从 Element Plus 默认后台壳升级为与项目主题一致的 PrimeVue 风格产品化框架
- 统一桌面端、平板端、移动端在导航、顶栏、内容承载层上的布局语言
- 为后续 T18/T19 页面级迁移提供稳定的视觉外壳和内容承载框架
- 顺带修正当前登录页、主布局相关中文文案乱码问题，统一为正常 UTF-8 中文表现

T17 不重写认证逻辑，不重做各业务页内部内容，仅重构“系统入口”和“主框架外壳”。

## 2. 设计定位

### 2.1 总体气质

本任务采用：**企业产品化为主，强化视觉层次但不过度展示化** 的方向。

视觉关键词：

- 现代企业级 SaaS
- 工业场景可信、稳重
- 毕业设计展示级完成度
- 清爽、有层次、但不花哨

这意味着：

- 保留后台系统应有的秩序感和可读性
- 避免过度营销化的落地页表达
- 在结构稳重的前提下，通过背景层次、圆角、阴影、留白和选中态强化“成品感”

### 2.2 与用户偏好的对齐

本设计基于已确认的偏好：

- 登录页采用 **居中单卡片**，不增加长篇产品介绍区
- 主布局采用 **产品化结构 + 更强视觉表达** 的组合方案
- 登录页与主布局都应明显摆脱当前 Element Plus 默认观感
- 视觉很重要，但不应破坏后台系统的专业和稳重

## 3. 视觉系统约束

### 3.1 色彩系统

T17 延续 T15/T16 的主题令牌体系，并在页面层明确采用以下语义倾向：

- **品牌深蓝**：`#0F4C81`
  - 用途：品牌标题、侧栏重点区、部分标题强调
- **交互主蓝**：`#1890FF`
  - 用途：主按钮、链接、焦点、激活态
- **页面背景底色**：`#F8FAFC`
  - 用途：应用背景、登录页背景主底
- **成功绿**：`#10B981`
- **警告橙**：`#F59E0B`

设计上采用“双层主色”策略：

- 深蓝负责可信度、秩序感和品牌稳定性
- 亮蓝负责主操作、反馈和交互聚焦

页面实现必须优先通过主题 token 或语义 class 消费这些颜色，不应在页面中直接散落硬编码色值。

### 3.2 表面、圆角与阴影

统一约束如下：

- 登录卡片圆角：**24px**
- 悬浮侧栏圆角：**16px**
- 主内容卡圆角：**20px ~ 24px**
- 控件高度：优先使用 T15/T16 已定义的大号控件高度，表单输入框在登录页以 **44px ~ 48px** 为目标

阴影原则：

- 使用大范围、低透明度、柔和边缘阴影
- 禁止使用短促、硬边、存在感过强的传统后台卡片阴影
- 页面层次优先依靠留白、边框、浅背景差异与柔和阴影共同建立

推荐阴影语义：

- 浮层/白卡：`0 10px 40px -10px rgba(0,0,0,0.08)` 量级
- 普通内容容器：更弱一级，不应抢主视觉

### 3.3 玻璃感边界

- 登录页允许引入轻度玻璃感/半透明卡片语言，以增强入口页完成度
- 主布局不采用玻璃态，保持后台系统稳定和清晰的结构表达

## 4. 登录页设计

### 4.1 结构定位

登录页采用 **单卡片居中** 方案，不新增左右分栏介绍区。

目标是：

- 打造进入系统时第一眼的品牌印象
- 维持界面聚焦，避免为毕业设计加入不必要的市场化宣传内容
- 在结构极简的情况下，通过背景层次、排版和表单质感实现高级感

### 4.2 背景层

背景采用浅灰蓝渐变底色，并叠加极轻的装饰层：

- 大面积浅灰蓝渐变
- 弱化光晕/柔光斑
- 很淡的网格感、颗粒感或工业秩序纹理

要求：

- 装饰层必须足够轻，不抢夺登录卡片焦点
- 背景是“有层次”，不是“有图案感”
- 不使用过强插画、照片、3D 物体或重品牌视觉噪音

### 4.3 登录卡片

登录卡片应包含：

1. 品牌标题区
2. 表单区
3. 辅助信息区
4. 页脚版权信息

卡片特点：

- 大圆角白色表面
- 轻玻璃感或高质感白卡效果
- 柔和高级阴影
- 内部留白充足
- 视觉重心垂直居中

### 4.4 文案规范

登录页文案统一为：

- 主标题：**工业配件供应链溯源系统**
- 副标题：**毕业设计展示 · 企业级业务后台原型**
- 用户名占位：**请输入用户名**
- 密码占位：**请输入密码**
- 记住登录：**记住登录状态**
- 主按钮：**登录系统**
- 页脚：`© 2026 工业配件供应链溯源系统`

关于演示账号提示：

- 不在卡片视觉主区域中高强度展示默认账号密码
- 若保留演示说明，仅作为卡片底部的低干扰辅助文案，例如：
  - **演示账号请联系管理员或查看项目说明**

### 4.5 表单与交互

登录表单保留现有业务逻辑：

- 用户名
- 密码
- 记住登录
- 提交 loading
- 回车提交
- 登录成功后按现有 redirect 逻辑跳转
- 登录失败时展示错误信息

交互要求：

- 输入框 focus ring 明确可见
- 主按钮 hover / active 有轻层次变化
- loading 态禁止重复提交
- 错误提示不能只依赖 toast；应保证用户在表单上下文中感知到失败
- 长错误文案不能破坏卡片布局

### 4.6 实现边界

- 优先复用 T16 已完成的 Prime 基础控件或 Base 兼容层
- 不再以 `el-card / el-input / el-button` 充当登录页视觉核心
- 保持登录业务逻辑与 store / router 行为不变
- 登录页相关文件需统一为 UTF-8 文本编码，消除中文乱码

## 5. 主布局设计

### 5.1 总体布局方向

主布局采用：**悬浮式圆角侧边栏 + 分层顶栏 + 白色大圆角内容容器**。

目标：

- 摆脱当前 Element Plus 默认后台骨架观感
- 让系统看起来像统一交付的现代后台产品
- 在不影响现有页面挂载逻辑的前提下，先把“壳”做稳定

### 5.2 桌面端布局

桌面端（建议 `>= 1024px`）采用以下结构：

- 页面外层统一浅灰蓝背景
- 左侧为悬浮侧栏，四周留出约 `16px` 外边距
- 侧栏宽度约 `248px ~ 264px`
- 右侧为主工作区，包含：
  - 顶栏区域
  - 主内容承载层

侧栏特点：

- 白底、圆角、轻阴影
- 顶部有品牌区
- 菜单项采用更清晰的间距和图标文字对齐
- 当前菜单采用胶囊状高亮，不使用传统大片底色条块

顶栏特点：

- 不是单薄的一条线性 header，而是具备层次的页面头部区域
- 左侧展示页面标题，可选带面包屑
- 右侧保留用户信息与扩展位（通知、头像等）

主内容承载层特点：

- 使用统一白色大圆角容器
- 作为路由页面的主要内容框架
- 提前兼容后续 T18/T19 页面迁移

### 5.3 平板端布局

平板端（建议 `768px ~ 1023px`）保持与桌面端一致的信息架构：

- 继续保留左侧栏
- 侧栏适度收窄至约 `224px`
- 页面间距略缩小
- 顶栏和内容区仍保持完整结构

原则：平板端应更像“紧凑桌面版”，而不是“提前变成手机布局”。

### 5.4 移动端布局

移动端（建议 `< 768px`）切换为抽屉导航：

- 顶栏保留菜单按钮、当前页标题、用户头像
- 左侧导航改为抽屉层从左侧滑出
- 内容区为单列滚动
- 间距缩小至 `12px ~ 16px`

要求：

- 页面切换后抽屉自动关闭
- 用户信息仍可在抽屉底部或顶栏中稳定识别
- 不因布局切换破坏导航、退出登录和权限显示逻辑

## 6. 布局组件边界

### 6.1 Login.vue

`frontend/src/shared/components/Login.vue` 继续保留为登录页入口文件，不强制拆出过多子组件。

原因：

- 登录页结构简单
- 逻辑集中，过度拆分收益低
- 便于在一个组件内统一控制视觉结构、提交逻辑和错误态

其职责应明确为：

- 管理登录表单状态与提交逻辑
- 渲染背景层与中心登录卡片
- 呈现登录页文案、辅助说明与版权信息
- 消费统一基础控件层

### 6.2 MainLayout.vue

`frontend/src/shared/components/layout/MainLayout.vue` 不应继续将桌面布局、移动布局、导航展示、顶栏结构、退出流程全部揉在一个超长模板中。

T17 推荐将其拆分为布局级子组件：

- **`MainLayout.vue`**
  - 负责导航数据、权限过滤、当前页面标题、移动端状态、抽屉状态、退出动作
- **`AppSidebarNav.vue`**
  - 负责桌面端悬浮侧栏
- **`AppTopbar.vue`**
  - 负责页面头部、标题、面包屑、用户操作区
- **`AppMobileNavDrawer.vue`**
  - 负责移动端抽屉导航
- **`AppContentFrame.vue`**（推荐）
  - 负责统一白色内容承载容器

### 6.3 组件拆分原则

- 仅拆布局结构，不拆业务逻辑
- `MainLayout.vue` 仍是单一状态编排入口
- 布局子组件尽量通过 props / emits 接口消费状态
- 导航项数据结构应保持单一来源，避免桌面端与移动端重复造结构
- 拆分的目标是让布局更清晰、后续页面迁移更容易，而非做形式化组件化

## 7. 交互与状态规范

### 7.1 导航交互

- 当前菜单：胶囊状高亮，颜色可使用浅蓝或浅灰蓝衬底
- hover：菜单项轻度浮起或背景增强
- icon 与文字同步进入选中态
- 动效控制在 `180ms ~ 220ms` 量级，不做夸张滑动或弹性效果

### 7.2 顶栏交互

- 页面标题比当前版本更明确
- 面包屑为辅助信息，权重低于页面主标题
- 用户信息区保留头像 + 用户名组合
- 为后续通知入口预留位置，但 T17 不强制接入真实通知能力

### 7.3 登录页状态

必须覆盖：

- 默认态
- focus 态
- loading 态
- 登录失败态
- 长错误文案展示
- 键盘回车提交

错误反馈要求：

- toast 可保留
- 但用户必须在登录表单上下文中感知到失败，而不是依赖瞬时全局提示

### 7.4 全局壳层状态

主布局应提前兼容以下页面承载状态：

- 空状态
- 加载骨架
- 长列表/长表单滚动
- toast 叠加
- 后端报错提示

T17 不要求重做这些页面内容，但布局外壳必须能够自然承接这些状态。

## 8. 技术与实现约束

### 8.1 与 T15/T16 的衔接

T17 必须建立在以下基础上：

- 使用 T15 定义的主题令牌语言
- 优先使用 T16 完成的 Prime 标准层与 Base 兼容层
- 视觉优先通过主题 token、共享控件和布局 class 表达
- 不在页面层重新发明一套局部风格系统

### 8.2 对 Element Plus 的约束

T17 的关键目标之一，是降低登录页与主布局对 Element Plus 默认骨架组件的视觉依赖。

因此：

- 不再以 `el-container / el-aside / el-header / el-main / el-menu` 作为主布局视觉核心
- 优先使用语义化 `div / nav / header / main` 与主题样式构建新骨架
- 若为过渡保留少量 Element Plus 能力，应仅作为功能依赖而非视觉主导

### 8.3 对 Tailwind / 自定义样式的约束

- 允许继续使用当前项目已有的原子类/样式组织方式
- 但页面层 class 命名与配色、圆角、阴影必须与 T15/T16 一致
- 不引入新的运行时 UI 框架，不新增 Tailwind 作为 T17 的前置依赖
- 用户提供的 AI Prompt、Tailwind 原型可作为视觉参考，但不作为运行时栈要求

### 8.4 编码与文案修复

- 登录页与主布局涉及的中文文案应全部恢复为正常中文
- 源文件保存编码统一为 UTF-8
- 若乱码来源于历史文件编码问题，应在实施时一并清理

## 9. 验收标准

### 9.1 登录页验收标准

T17 完成后，登录页应满足：

- 视觉上明显不同于当前普通 Element 登录卡片
- 保持单卡片居中结构
- 背景存在轻层次但不过度装饰
- 品牌标题、副标题、版权信息完整
- 中文文案无乱码
- 登录成功、失败、记住登录、回车提交逻辑保持可用
- 桌面端与移动端不破版

### 9.2 主布局验收标准

T17 完成后，主布局应满足：

- 桌面端形成“悬浮侧栏 + 分层顶栏 + 白色内容容器”的稳定结构
- 当前菜单为明显但克制的胶囊高亮
- 顶栏信息层次优于当前版本
- 移动端抽屉导航可正常使用
- 内容区间距、圆角、阴影统一
- 不再依赖 Element Plus 默认容器骨架充当主视觉

### 9.3 一致性验收标准

- 登录页与主布局视觉属于同一产品体系
- 颜色、圆角、阴影、控件态与 T15/T16 一致
- 整体观感应接近“已交付后台系统”，而不是课程作业页面的零散拼接

## 10. 范围边界

### 10.1 T17 包含

- 登录页视觉重构
- 主布局结构与视觉重构
- 布局层适度拆分子组件
- 登录页与主布局中文文案/乱码修复
- 响应式行为与基础交互状态统一

### 10.2 T17 不包含

- 不重写认证接口、权限接口或登录业务协议
- 不重做业务页内部表格、表单、图表与详情内容
- 不在本任务中推进 T18/T19 的页面级全面换皮
- 不新增复杂品牌介绍页、落地页、宣传区模块
