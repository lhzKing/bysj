# 工业企业配件供应链溯源系统数据库设计

## 1. 设计目标

数据库围绕“全链路可追溯、状态可快速查询、权限可精细控制、数据可验真防篡改”四个目标设计：

- 以**事实表 + 快照表**组合兼顾写入审计与查询性能。
- 以 **RBAC 权限模型**支撑多角色协同操作。
- 以 **Hash 链 + RSA 签名字段**支撑日志级防篡改验证。
- 以 **乐观锁版本号**保障并发扫码时状态更新一致性。

---

## 2. 总体ER关系（文字版）

核心实体及关系如下：

1. **sys_role（角色）** 与 **sys_user（用户）**：一对多
   - 一个角色可绑定多个用户。
2. **sys_role（角色）** 与 **sys_permission（权限）**：多对多
   - 通过中间表 **sys_role_permission** 建立关联。
3. **base_part_spec（配件规格/SPU）** 与 **trace_snapshot（溯源快照）**：一对多
   - 一个配件规格可对应多条溯源码实例。
4. **trace_snapshot（溯源快照）** 与 **trace_lifecycle_log（生命周期日志）**：一对多
   - 快照记录“当前状态”，日志记录“全量历史事件”。
5. **trace_lifecycle_log（生命周期日志）** 自关联：
   - 字段 `correction_of` 外键指向同表 `id`，用于红冲蓝补/修正链路。

## 3. 表结构设计

> 以下字段来自 `backend/sql/init_schema.sql` 与当前后端实体实现。

### 3.1 sys_role（角色表）

**用途**：定义系统角色（超级管理员、系统管理员、生产人员、仓库人员、物流人员、普通用户）。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| role_code | VARCHAR(32) | UNIQUE, NOT NULL | 角色代码（如 SUPER_ADMIN） |
| role_name | VARCHAR(64) | NOT NULL | 角色名称 |
| remark | VARCHAR(255) |  | 备注 |
| create_time | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |

---

### 3.2 sys_permission（权限表）

**用途**：定义权限点，包括接口方法、路径模式和权限语义。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| perm_code | VARCHAR(64) | UNIQUE, NOT NULL | 权限代码（如 trace:create） |
| perm_name | VARCHAR(64) | NOT NULL | 权限名称 |
| api_method | VARCHAR(10) |  | HTTP 方法（GET/POST/.../*） |
| api_pattern | VARCHAR(128) |  | API 路径模式（支持 `*`） |
| remark | VARCHAR(255) |  | 备注 |
| create_time | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |

---

### 3.3 sys_role_permission（角色权限关联表）

**用途**：建立 RBAC 多对多关系。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| role_id | BIGINT | PK(联合), FK | 角色ID |
| permission_id | BIGINT | PK(联合), FK | 权限ID |

**约束**：
- `fk_rp_role`: `role_id -> sys_role.id`（ON DELETE CASCADE）
- `fk_rp_perm`: `permission_id -> sys_permission.id`（ON DELETE CASCADE）

---

### 3.4 sys_user（用户表）

**用途**：保存登录主体、角色绑定与安全状态。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| username | VARCHAR(64) | UNIQUE, NOT NULL | 用户名 |
| password | VARCHAR(128) | NOT NULL | BCrypt 密文密码 |
| role_id | BIGINT | NOT NULL, FK | 角色ID |
| token_version | INT | NOT NULL, DEFAULT 0 | Token版本号（改密/改角色递增） |
| status | TINYINT | NOT NULL, DEFAULT 1 | 1=启用，0=禁用 |
| create_time | DATETIME | DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**设计要点**：
- `token_version` 与 JWT 联动，实现“全量旧 token 失效”。
- `status` 实现用户启停控制。

---

### 3.5 base_part_spec（配件规格表，SPU）

**用途**：管理配件标准信息，用于赋码时绑定基础规格。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | SPU主键 |
| part_code | VARCHAR(64) | UNIQUE, NOT NULL | 规格编码 |
| part_name | VARCHAR(128) | NOT NULL | 配件名称 |
| part_type | VARCHAR(64) | NOT NULL | 配件类别 |
| model | VARCHAR(64) |  | 型号 |
| manufacturer | VARCHAR(128) |  | 制造商 |
| unit | VARCHAR(32) |  | 计量单位 |
| remark | VARCHAR(255) |  | 备注 |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

---

### 3.6 trace_snapshot（溯源快照表）

**用途**：保存每个溯源码实例的当前状态，支撑高频查询。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| trace_code | VARCHAR(64) | PK | 溯源码 |
| spu_id | BIGINT | NOT NULL | 关联SPU |
| current_status | VARCHAR(32) | NOT NULL | 当前状态 |
| current_node | VARCHAR(64) |  | 当前节点 |
| current_owner | VARCHAR(64) |  | 当前责任方 |
| province | VARCHAR(32) |  | 省份（地图统计） |
| city | VARCHAR(32) |  | 城市 |
| last_event_time | DATETIME | NOT NULL | 最新业务时间 |
| last_log_id | BIGINT | NOT NULL | 最新日志ID |
| last_hash | CHAR(64) | NOT NULL | 链尾哈希（下一条 prev_hash） |
| version | INT | NOT NULL, DEFAULT 0 | 乐观锁版本号 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP | 更新时间 |

**设计要点**：
- 使用 `version`（MyBatis-Plus `@Version`）防并发覆盖。
- `last_hash` 将快照与日志链连接，保障链式连续性。

---

### 3.7 trace_lifecycle_log（溯源生命周期日志表）

**用途**：记录每次流转事件的全量事实，是审计、验链、追责的核心表。

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| id | BIGINT | PK, AUTO_INCREMENT | 日志主键 |
| trace_code | VARCHAR(64) | NOT NULL | 溯源码 |
| spu_id | BIGINT | NOT NULL | 关联SPU |
| action_type | VARCHAR(32) | NOT NULL | 事件类型（INIT/INBOUND/OUTBOUND/TRANSFER/EXCEPTION/CORRECTION） |
| from_node | VARCHAR(64) |  | 上游节点 |
| to_node | VARCHAR(64) |  | 下游节点 |
| province | VARCHAR(32) |  | 省份 |
| city | VARCHAR(32) |  | 城市 |
| event_time | DATETIME | NOT NULL | 业务时间（Event Time） |
| ingest_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 入库时间（Ingest Time） |
| prev_hash | CHAR(64) | NOT NULL | 前一条日志哈希 |
| current_hash | CHAR(64) | NOT NULL | 当前日志哈希 |
| correction_of | BIGINT | FK(NULL) | 被修正的原日志ID |
| operator | VARCHAR(64) | NOT NULL | 操作人 |
| signature | VARCHAR(512) |  | RSA数字签名（Base64） |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |

**索引设计**：
- `idx_trace_code(trace_code)`：溯源码历史查询
- `idx_event_time(event_time)`：时间窗统计
- `idx_ingest_time(ingest_time)`：写入时间检索
- `idx_province(province)`：地图聚合
- `idx_from_to(from_node, to_node)`：拓扑关系
- `idx_correction_of(correction_of)`：修正链追踪

**约束**：
- `fk_correction_of`：`correction_of -> trace_lifecycle_log.id`

## 4. 安全与一致性设计说明

### 4.1 防篡改机制

每条日志保存：
- `prev_hash`：上一条日志哈希
- `current_hash`：当前日志哈希
- `signature`：对标准化拼接数据的 RSA 签名

系统可对链进行三类校验：
1. 链连续性校验（`prev_hash` 是否连续）
2. 哈希重算校验（`current_hash` 是否一致）
3. 签名验证（`signature` 是否可通过公钥验证）

### 4.2 并发控制机制

- `trace_snapshot.version` 启用乐观锁。
- 流转更新使用独立事务 + 失败重试策略。
- 冲突超限后返回并发冲突错误，避免脏写。

### 4.3 权限控制机制

- `sys_user` + `sys_role` + `sys_permission` + `sys_role_permission` 构成标准 RBAC。
- 支持权限继承（如 `trace:inbound -> trace:scan -> trace:view`）。
- 支持细粒度动作权限（入库/出库/流转分离）。

## 5. 典型业务数据流与表写入

### 5.1 生产赋码

1. 写入 `trace_lifecycle_log`（INIT，prev_hash=GENESIS，生成hash+signature）
2. 写入 `trace_snapshot`（当前状态=INIT，last_log_id、last_hash）

### 5.2 扫码流转

1. 读取 `trace_snapshot` 当前版本
2. 插入 `trace_lifecycle_log` 新事件
3. 更新 `trace_snapshot` 当前状态/节点/链尾哈希（带 version 条件）
4. 冲突则按策略重试

### 5.3 验链查询

1. 按 `trace_code` 拉取完整日志链
2. 逐条执行链连续、hash重算、签名验证
3. 返回验证结果与错误明细

## 6. 规范与优化建议

1. **外键补强建议**：`trace_snapshot.spu_id`、`trace_lifecycle_log.spu_id`、`trace_lifecycle_log.trace_code`可按业务压力评估是否补充外键或逻辑约束。
2. **归档策略**：日志表数据量增长快，可按月分区或冷热分层。
3. **索引维护**：统计查询与写入冲突时，定期评估二级索引数量。
4. **密钥治理**：生产环境固定管理 RSA 私钥，禁用自动生成。
5. **审计扩展**：可新增操作终端、IP、请求ID等字段，提升取证能力。

---

## 7. 小结

本数据库方案采用“权限域 + 配件域 + 溯源事实域”分层建模，既支持工业供应链场景下的高并发流转记录，又能通过哈希链与数字签名确保数据可验真、可追责。整体结构与当前系统实现一致，具备较强工程实用性与后续扩展空间。
