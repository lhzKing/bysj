## §1 扫码弹窗的来源节点/目标仓库全靠手填

> **2026-05-11 老师指正：** 表述需更精确——后端实际不是"完全没做运单"，而是做成了**两条独立的扫码链路**，普通弹窗那条没接运单上下文；后端节点绑定也不只校验、还有部分预填能力，但被前端必填校验挡死了。本节按指正重写。

### 1.1 现象

截图所示"入库登记"弹窗 6 个字段 5 个需要操作员手填：来源节点、目标仓库、省份、城市、入库时间、备注。出库 / 物流流转复用同一表单，只是 label 文案不同。

### 1.2 项目里实际有两条扫码链路

```
普通扫码链路（你看到的"扫码工位"）：
  ScanHub → /api/traces/{code}/available-actions
         → ScanFlowDialog（必填 fromNode/toNode/province/city）
         → POST /api/traces/{code}/events
  特点：要求前端填 fromNode / toNode / province / city

任务扫码链路（"仓库/物流任务工作台"，前端已有独立 view）：
  TraceFlowTaskWorkbench → POST /api/trace-flow-tasks/{id}/scan
  特点：任务 已知 sourceNode / targetNode，前端只传 traceCode + eventTime，后端自动构造 ScanTraceRequest
```

**也就是说：**
- 后端**不是**"运单完全没做"，而是任务扫码闭环已经做完
- 前端**不是**"完全没接"，TraceFlowTaskWorkbench.vue 已经接了任务接口
- **真正缺的**是：**普通扫码工位（ScanHub + ScanFlowDialog）尚未接入运单上下文**，扫码时不会自动匹配开放运单、不会用任务的 source/target 推导节点

| 动作 | 缺什么 | 后端补齐策略 |
|---|---|---|
| INBOUND | 没传 toNode | 用用户**唯一/默认**绑定节点补齐 |


后端的预填能力被前端校验挡死，永远走不到。**要解锁后端补齐，前端只需把这 4 个 if 拆掉**——让空字段直接发到后端，后端会自动补齐。

## §A.3 单品码列表列宽错乱 + 移动端按钮单字符竖排 ✅

**根因 1**：[`TraceAssignmentWorkbench.vue`](frontend/src/features/trace/views/TraceAssignmentWorkbench.vue) 里 `assignment__row` 这个 class **被两套规则同名定义**——表单 row 是 `display: grid; grid-template-columns: 1fr 1fr`，给表格 `<tr>` 误用同名 class，CSS cascade 让 `<tr>` 变成两列网格，6 个 `<td>` 拆成 2×3 错位。

**根因 2**：右侧"当前选中"卡用 grid 280px 占位 + 原 `<td>` 上挂 `display: flex` 把单元格踢出 table-cell layout，列宽分配再次错乱。

**根因 3**（移动端）：`table-layout: fixed` + 6 列固定宽度（48+230+96+76+120 = 570px），640px viewport 减 padding 后 actions 列只剩 ~80px，按钮文本被压成单字一行。
